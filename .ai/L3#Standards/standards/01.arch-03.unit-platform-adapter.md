# 单位平台适配器规范

> **级别**：强制
> **创建日期**：2026-05-18

---

## 核心原则

NACP 的单位平台适配层必须遵守：

```text
平台 adapter 独立，协议级工具复用。
```

也就是：

1. 已经明确有独立名称、独立项目、独立分支、独立接口特征的平台，必须独立成目录和 adapter。
2. 不因为当前接口暂时兼容母类型，就把平台糅合到母类型目录中。
3. 不使用“继承式混写”表达平台关系，例如不要把 `rixapi`、`shellapi` 的业务逻辑直接写进 `newapi`。
4. 可以复用协议级基础设施，例如 HTTP client、URL normalize、JSON decode、WAF/Cloudflare 判断、凭据脱敏、标准 `/v1/models` 解析。
5. 平台业务能力必须由各自 adapter 显式声明和实现，包括识别、账号信息、余额、分组、令牌、模型、公告、签到、OAuth/session 刷新等。

判断边界：

```text
能代表平台差异的，不复用。
不能代表平台差异的，复用。
```

---

## 目录规范

单位平台差异统一放在：

```text
service/unitplatform/
```

平台目录按具体平台拆分：

```text
service/unitplatform/
  types.go
  registry.go
  detect.go
  http.go
  credentials.go
  capabilities.go

  common/
    panel.go
    models.go
    quota.go
    html.go

  newapi/
  rixapi/
  shellapi/
  oneapi/
  oneapifork/
  veloera/
  onehub/
  donehub/
  anyrouter/
  sub2api/
  openai/
  claude/
  gemini/
  geminicli/
  antigravity/
  cliproxyapi/
  codex/
```

`common/` 不是平台类型，只能放协议级 helper。任何带平台语义的判断、字段、接口和换算规则都必须放回具体平台目录。

## 当前已识别平台分类

| 平台类型 | 来源 | 说明 |
|---|---|---|
| `newapi` | NewAPI-family | NewAPI 兼容面板 |
| `rixapi` | NiceApiManager / NewAPI fork | RixAPI 独立识别和用户头 |
| `shellapi` | NiceApiManager / NewAPI fork | ShellAPI 独立状态字段和响应头 |
| `oneapi` | Metapi | One API 原版 |
| `oneapifork` | NACP 当前兼容类型 | 未能进一步识别的 One API 系分支 |
| `veloera` | Metapi | 额度换算和用户头不同 |
| `onehub` | Metapi | OneHub 独立模型 fallback |
| `donehub` | Metapi | DoneHub 余额语义不同，不支持签到 |
| `anyrouter` | Metapi | AnyRouter 独立平台名 |
| `sub2api` | Metapi / sub2api | JWT、订阅、keys、groups 接口独立 |
| `openai` | Metapi | API key 直连供应商 |
| `claude` | Metapi | Anthropic API key 直连供应商 |
| `gemini` | Metapi | Gemini native / OpenAI-compatible 双模式 |
| `geminicli` | Metapi | Google Cloud Code / Gemini CLI |
| `antigravity` | Metapi | Antigravity OAuth 模型接口 |
| `cliproxyapi` | Metapi | CLIProxyAPI |
| `codex` | Metapi | Codex OAuth/session 平台 |

新增平台时，只要它有明确名称、明确接口特征、明确上游项目或明确运营意义，就新增独立目录，不合并进母类型。

当前 OAuth/CLI 直连平台的边界：

| 平台类型 | 识别/读取方式 | 能力边界 |
|---|---|---|
| `codex` | `GET <base>/models?client_version=1.0.0`，`Originator: codex_cli_rs`，可选 `Chatgpt-Account-Id` | 只读取模型；不支持余额快照、平台令牌读取、平台令牌创建 |
| `geminicli` | 校验 Google Cloud 项目是否启用 `cloudaicompanion.googleapis.com`，通过后返回 Gemini CLI 静态模型清单 | 只读取模型；账号 ID 表示 Google Cloud 项目 ID；不支持余额快照和令牌管理 |
| `antigravity` | `POST /v1internal:fetchAvailableModels`，按单位 API 地址、默认、daily、sandbox 顺序尝试 | 只读取模型；账号 ID 可作为 project；不支持余额快照和令牌管理 |

## Adapter 必须表达的能力

每个平台 adapter 至少需要声明：

1. `Type`：平台类型字符串。
2. `Detect`：识别逻辑。
3. `Capabilities`：平台能力。
4. `CredentialSpec`：需要哪些凭据。
5. `FetchSnapshot`：能否采集账号快照。
6. `FetchTokens`：能否读取平台令牌。
7. `CreateToken`：能否创建平台令牌。
8. `FetchTokenOptions`：能否读取模型和分组。
9. `FetchModels`：能否读取模型。

不支持的能力必须明确返回“不支持”，不能假装成功或返回伪数据。

## 复用限制

允许复用：

- HTTP client、timeout、proxy、TLS 基础处理。
- URL 标准化。
- `common.Marshal` / `common.Unmarshal` / `common.DecodeJson` JSON 包装。
- 响应体大小限制。
- WAF/Cloudflare challenge 判断。
- 凭据脱敏。
- 标准 OpenAI-compatible `/v1/models` 响应解析。

禁止复用或混写：

- 平台识别字段。
- 平台专用 Header。
- 平台余额换算。
- 平台令牌创建 payload。
- 平台分组接口 fallback 顺序。
- 平台 OAuth/session 刷新。
- 平台公告、订阅、充值倍率等业务字段解析。

## 兼容入口

旧入口可以短期保留，但必须代理到新适配层：

```text
service/unitdetect  -> service/unitplatform
service/unitmonitor -> service/unitplatform
```

最终目标是：

```text
controller/unit.go
controller/unit_monitor.go
service/unit_monitor.go
service/unit_tokens.go
        ↓
service/unitplatform/<platform>/
```

---

## 验收标准

1. 单位类型识别不再把已知平台糅合为母类型。
2. 单位详情、账号监控、令牌管理读取同一套平台能力声明。
3. 平台不支持的功能在前端显示为不可用或明确提示。
4. 新增平台只新增目录和注册，不修改其他平台业务逻辑。
5. 所有 JSON 处理继续使用 `common/json.go` 包装。
6. 数据库变更继续遵守三库兼容规范。
