# NACP 变更验证与上线保障体系（CAF）

> **英文名**：NACP Change Assurance Framework
> **简称**：CAF
> **版本**：v1.0.0
> **创建日期**：2026-05-15

---

## 一、定位

CAF 不是单个测试计划，也不是某次发布 checklist。

CAF 是 NACP 面向所有新增、增强、修复、重构的标准化质量保障体系：

```text
系统理解 -> 影响分析 -> 测试计划 -> 执行验证 -> 证据留存 -> 上线判断 -> 知识回流
```

目标是让 NACP 团队比 NewAPI 官方更懂 NewAPI：

1. 吃透 NewAPI 原版模块、数据、配置、后台任务、前端页面、API。
2. 吃透 NACP 每次增强对原系统的影响。
3. 每次变更都能标准化生成测试范围。
4. 尽量降低遗漏未测试模块的概率。
5. 每次测试结论都能回流到 `.ai`，让下一次更准确。

---

## 二、适用范围

所有 NACP 变更都必须套用 CAF，至少完成影响分析。

| 变更类型 | 是否必须走 CAF |
|----------|----------------|
| 新功能 | 必须 |
| 功能增强 | 必须 |
| Bug 修复 | 必须 |
| 日志/统计/计费相关调整 | 必须 |
| 数据库迁移 | 必须 |
| 前端页面改动 | 必须 |
| 部署脚本改动 | 必须 |
| 纯文档改动 | 可简化，只做命名和引用检查 |

## 三、CAF 七阶段

### Phase 1：系统知识定位

先定位本次变更涉及哪些系统：

```text
Relay / Channel / Billing / Quota / Token / User / Admin / Logs / Stats / Frontend / DB / Cache / Deployment
```

必须优先读取：

1. 对应 L3 规范。
2. 对应 L5 知识文档。
3. NewAPI 原版相关源码。
4. NACP 已改动代码。
5. 相关测试和线上验证记录。

### Phase 2：影响分析树

每次变更必须输出影响树。

基础模板：

```text
变更点
├── 直接影响模块
├── 间接影响模块
├── 数据影响
├── 计费影响
├── 日志/统计影响
├── 权限/安全影响
├── 前端展示影响
├── 线上真实渠道影响
└── 回滚/兼容影响
```

### Phase 3：风险分级

| 风险等级 | 定义 | 测试要求 |
|----------|------|----------|
| P0 | 影响请求主链路、计费、扣费、余额、安全、部署 | 必须本地 + 线上真实链路 + 数据验证 |
| P1 | 影响管理后台、日志、统计、配置、渠道选择 | 必须集成测试 + 前端验证 + 数据检查 |
| P2 | 局部展示、低风险配置、非核心工具 | 冒烟 + 回归 |
| P3 | 文档、注释、无行为变化 | 静态检查 |

### Phase 4：测试计划生成

测试计划必须包含：

1. 变更摘要。
2. 影响树。
3. 风险矩阵。
4. 测试环境。
5. 测试数据。
6. 测试用例表。
7. 通过标准。
8. 失败排查入口。
9. 回滚条件。
10. 验收证据。

### Phase 5：执行验证

CAF 执行验证必须按环境逐级推进：

```text
本地开发环境测试 -> 线上测试服务器测试 -> 正式服务器部署后观察
```

原则：

1. 本地测试没有通过，不允许进入线上测试服务器。
2. 线上测试服务器没有通过，不允许部署正式服务器。
3. 正式服务器只做部署后观察和必要冒烟，不用于首次发现核心问题。
4. P0/P1 变更必须同时留下本地证据和线上测试服务器证据。
5. 任一阶段发现 P0/P1 问题，必须回到本地修复并重新开始本地验证。

#### Stage A：本地开发环境测试

目的：在最短反馈周期内发现代码、接口、数据结构、前端显示和自动化测试问题。

最低要求：

1. 运行相关单元测试、集成测试或专项测试。
2. 本地启动后端和前端，优先用 API 验证核心路径。
3. 对日志、计费、统计、trace 类功能检查接口字段和必要数据库原始行。
4. 前端页面只作为显示层验证，不能替代 API 证据。
5. 本地发现的问题必须先在本地修复并补测试。

#### Stage B：线上测试服务器测试

目的：在接近真实部署和真实渠道的环境中验证行为。

最低要求：

1. 通过标准部署流程把当前版本部署到测试服务器。
2. 使用测试账号、测试 token、测试渠道执行真实请求。
3. 验证请求结果、日志链路、计费、余额、统计、渠道健康状态。
4. 验证测试服务器数据库和环境变量没有与本地假设冲突。
5. 记录测试服务器版本、镜像、commit、关键 API 响应和日志证据。

#### Stage C：正式服务器部署后观察

目的：确认正式环境部署成功，并观察是否出现回归。

最低要求：

1. 只在本地和测试服务器都通过后执行正式部署。
2. 部署后做低风险冒烟，不主动制造破坏性故障。
3. 观察错误率、渠道异常、计费、余额、日志、统计。
4. 若出现 P0/P1 异常，优先回滚或止血，再进入本地复现和修复。

#### 分层验证内容

按风险等级执行：

| 层级 | 内容 |
|------|------|
| L0 静态检查 | diff 范围、命名、无无关文件、无保护信息变更 |
| L1 单元测试 | 纯函数、状态机、格式化、边界条件 |
| L2 集成测试 | DB、API、缓存、权限、迁移 |
| L3 Relay E2E | OpenAI/Claude/Gemini/AWS 等真实请求路径 |
| L4 前端验证 | 页面、筛选、展开、交互、移动端/桌面 |
| L5 线上真实渠道 | 测试站真实 token、真实渠道、真实日志 |
| L6 回归矩阵 | 受影响模块和邻近模块回归 |
| L7 发布观察 | 部署后日志、错误率、计费、统计观察 |

#### API 先行原则

涉及后台、日志、统计、计费、渠道、用户、token 的问题，必须先用 API 和数据字段验证，再用浏览器页面确认展示。

原因：

1. 前端页面可能经过筛选、折叠、隐藏、分页、缓存和本地状态处理。
2. 浏览器截图只能证明展示结果，不能证明底层数据是否正确。
3. 日志页、trace、计费、统计类问题必须能追溯到接口响应和数据库字段。

最低检查顺序：

```text
1. /api/status
2. 登录或复用管理会话
3. 直接调用目标 API
4. 检查关键字段和分页 total
5. 必要时检查数据库原始行
6. 再打开前端页面验证显示层
```

日志页面专项要求：

1. 先查 `/api/log/grouped`，确认 `items`、`total`、`type`、`trace_role`、`request_id`、`trace_id`、`trace_seq`。
2. 再查 `/api/log/trace?request_id=...`，确认链路步骤、最终 consume/error_visible、计费字段。
3. 再查 `/api/log/traces`，确认同一 `request_id` 只能有一条摘要。
4. 后台健康探测、系统任务、异步补偿等非用户请求日志，不得污染默认用户/管理员日志列表。
5. 如果页面为空，不能先判定前端坏；必须先确认 API 第一页是否被隐藏类数据占满。

### Phase 6：验收证据

每次测试必须留下证据：

1. 命令输出摘要。
2. API 响应摘要。
3. 数据库关键查询结果。
4. 管理员日志截图或字段摘要。
5. 用户侧结果。
6. 计费/余额变化。
7. 未覆盖风险。

### Phase 7：知识回流

测试完成后必须判断是否需要更新：

1. `L3#Standards (Standards)`：新增长期规则。
2. `L5#Knowledge (Knowledge)`：新增系统知识、测试经验。
3. `L4#Changelog (Changelog)`：记录本次过程。
4. 未来自动化测试脚本。

## 四、强制检查域

任何影响核心链路的变更，都必须逐项考虑：

| 检查域 | 必查问题 |
|--------|----------|
| 请求链路 | 请求是否能到达正确 handler/adaptor/upstream |
| 渠道选择 | 优先级、权重、禁用、健康状态、亲和、auto 分组是否正确 |
| 计费 | 预扣费、结算、退款、违规扣费、订阅抵扣是否正确 |
| 计量 | prompt/completion/cache/audio/image/task token 是否正确 |
| 额度 | 用户余额、分组倍率、模型倍率、渠道倍率是否正确 |
| 日志 | `logs.type`、`trace_*`、`other`、管理员信息是否正确 |
| 统计 | dashboard、usage、channel stats 是否被污染 |
| 权限 | 普通用户、管理员、token 权限是否隔离 |
| 前端 | 列表、详情、筛选、复制、展开、i18n 是否正确 |
| 数据库 | SQLite/MySQL/PostgreSQL 是否兼容 |
| 缓存 | Redis/in-memory 是否一致，缓存失效是否正确 |
| 部署 | Docker、gogogo、环境变量、迁移是否可重复执行 |

## 五、CAF 输出物命名

测试计划和执行记录放在 `L5#Knowledge (Knowledge)`：

```text
caf-<变更主题>-plan.md
caf-<变更主题>-report.md
caf-<变更主题>-checklist.md
```

示例：

```text
caf-smart-failover-trace-plan.md
caf-smart-failover-trace-report.md
caf-log-trace-grouping-checklist.md
```

## 六、最小通过标准

一次变更只有满足以下条件，才能视为 CAF 通过：

1. 已明确影响树。
2. 已覆盖直接影响模块。
3. 已覆盖至少一层邻近模块回归。
4. P0/P1 变更已验证线上测试站。
5. 计费、日志、统计没有明显污染。
6. 失败用例有明确结论，不被忽略。
7. 剩余风险被记录。
8. 相关知识已回流 `.ai`。
