# NACP AI 记忆体系命名规范

> **版本**：v1.0.0
> **创建日期**：2026-05-15
> **依据**：`ZERO/.ai/L3#Standards/standards/10.ai-memory-02.naming.md`、`ZERO/.ai/L3#Standards/standards/10.ai-memory-03.six-layer-naming.md`

---

## 一、核心结论

NACP 的 `.ai` 目录已经采用 ZERO 六层架构目录：

```text
.ai/L0#Execution/
.ai/L1#Overview/
.ai/L2#Index/
.ai/L3#Standards/
.ai/L4#Changelog/
.ai/L5#Knowledge/
```

但当前文件命名还没有完全按 ZERO 的分类编号和 AI 友好命名体系收敛，尤其是：

1. `L3#Standards (Standards)` 中存在 `02.dev-*`、`03.quality-*` 这类非 ZERO 分类编号。
2. `L5#Knowledge (Knowledge)` 中多数文档是自由标题式命名，缺少统一领域前缀和文档类型后缀。
3. `L2#Index (Index)` 还没有同步收录 AI 记忆体系命名规范。
4. 部分文件名过短，不能从文件名直接判断文档定位、系统域和用途。

因此，从 2026-05-15 起，NACP `.ai` 新增或迁移文档必须遵守本文档。

---

## 二、ZERO 基础规则

### 2.1 目录命名

必须使用：

```text
L#英文名
```

允许：

```text
L0#Execution
L1#Overview
L2#Index
L3#Standards
L4#Changelog
L5#Knowledge
```

禁止：

```text
L0
L0-Execution
L0_Execution
L0#工作执行
```

### 2.2 通用文件名规则

除固定入口文件 `README.md` 外：

1. 文件名使用小写。
2. 单词之间用连字符。
3. 不使用下划线。
4. 不使用中文。
5. 不使用空格。
6. 不为了简短省略关键语义。
7. 文件名应能表达“领域 + 主题 + 文档类型”。

## 三、L3#完整规范 (Standards) 命名

### 3.1 格式

```text
{大类编号}.{类别}-{序号}.{主题}.md
```

示例：

```text
01.arch-01.smart-failover-trace.md
02.backend-01.json-wrapper-usage.md
06.quality-01.git-workflow.md
06.quality-02.change-assurance-framework.md
10.ai-memory-01.naming.md
```

### 3.2 分类编号

| 编号 | 类别 | 用途 |
|------|------|------|
| `00` | `meta` | 元规范、规范治理 |
| `01` | `arch` | 架构、系统设计、核心链路 |
| `02` | `backend` | Go 后端、数据库、DTO、Relay 后端逻辑 |
| `03` | `frontend` | React 前端、页面、组件、交互 |
| `04` | `data` | 数据结构、知识图谱、数据模型 |
| `05` | `biz` | 业务规则、运营规则 |
| `06` | `quality` | 测试、发布、Git、评审、验收 |
| `07` | `security` | 鉴权、安全、密钥、权限 |
| `08` | `workflow` | 工作流、协作流程、Agent 流程 |
| `09` | `tool` | 工具、脚本、部署工具 |
| `10` | `ai-memory` | `.ai` 体系、知识沉淀、命名、索引 |

### 3.3 L3 文件迁移记录

| 原文件 | 问题 | 当前文件 |
|----------|------|--------------|
| `01.arch-01.relay-retry.md` | 名称偏旧，未体现 SFT 正式命名 | `01.arch-01.smart-failover-trace.md` |
| `01.arch-02.baseline.md` | 语义过短 | `01.arch-02.newapi-v0-13-2-baseline.md` |
| `02.dev-01.json-usage.md` | `dev` 不是 ZERO 分类 | `02.backend-01.json-wrapper-usage.md` |
| `02.dev-02.db-compat.md` | `dev` 不是 ZERO 分类，语义可更完整 | `02.backend-02.database-compatibility.md` |
| `02.dev-03.dto-pointer.md` | `dev` 不是 ZERO 分类，语义可更完整 | `02.backend-03.relay-dto-explicit-zero-values.md` |
| `03.quality-01.git.md` | `quality` 应使用 `06` 类别 | `06.quality-01.git-workflow.md` |

这些文件已在 2026-05-15 执行第一批 L3 命名迁移。后续新增 L3 文件必须直接使用目标命名格式。

迁移要求：

1. 使用 `git mv`。
2. 同步更新 `L2#Index (Index)`。
3. 同步更新所有 `.ai` 内部引用。
4. 增加一条 `L4#Changelog (Changelog)` 记录。

## 四、L4#操作日志 (Changelog) 命名

### 4.1 格式

```text
YYYY-MM-DD-<short-event-name>.md
```

要求：

1. 日期使用本地时区日期。
2. 事件名小写连字符。
3. 事件名必须能表达变更主题。
4. Changelog 是历史事实，不强制重命名旧记录，除非旧文件名严重误导。

示例：

```text
2026-05-15-ai-memory-naming-standard.md
2026-05-15-smart-failover-trace-log-compatibility.md
```

## 五、L5#知识图谱 (Knowledge) 命名

ZERO 原始框架中，`L5#Knowledge (Knowledge)` 更偏向图数据库/MCP 语义搜索，不强制直接存放文件。NACP 当前把 L5 用作长期知识文档存储，因此需要新增 NACP 本地约定。

### 5.1 格式

```text
{领域}.{主题}.{文档类型}.md
```

但为了与 ZERO “同一文件名内不混用 `-` 和 `.`”的原则兼容，NACP L5 实际使用：

```text
{领域}-{主题}-{文档类型}.md
```

### 5.2 领域前缀

| 前缀 | 用途 |
|------|------|
| `newapi` | NewAPI 原版源码、机制、模块图谱 |
| `nacp` | NACP 自身增强、业务需求、变更说明 |
| `sft` | NACP 智能容错链路（Smart Failover Trace） |
| `caf` | NACP 变更验证体系（Change Assurance Framework） |
| `billing` | 计费、计量、额度、价格 |
| `channel` | 渠道、亲和、健康、选择策略 |
| `usage-log` | 日志、统计、链路展示 |
| `test` | 测试计划、执行记录、回归方案 |
| `data` | 数据分析、迁移分析、数据结构 |

### 5.3 文档类型后缀

| 后缀 | 用途 |
|------|------|
| `analysis` | 源码/机制分析 |
| `map` | 系统图谱、模块关系 |
| `plan` | 计划 |
| `playbook` | 可反复执行的操作手册 |
| `checklist` | 检查清单 |
| `report` | 执行结果报告 |
| `decision` | 决策记录 |
| `review` | 评审材料 |
| `changelog` | 面向评审的变更汇总 |

## 六、L2#规范索引 (Index) 更新规则

当 L3 新增、删除、重命名规范时，必须同步更新：

```text
.ai/L2#Index/toc.md
```

更新内容包括：

1. 文件名。
2. 说明。
3. 分类位置。
4. 当前重点。
5. 参考资料。

---

## 七、迁移顺序

命名迁移必须分阶段，避免引用断裂。

### Phase 1：建立规则

1. 新增本文档。
2. 更新 L2 索引。
3. 增加 L4 changelog。

### Phase 2：迁移 L3

1. `git mv` L3 文件。
2. 更新 `.ai` 内部引用。
3. 检查 `rg` 不再出现旧路径；迁移记录表中保留旧文件名作为历史对照例外。

### Phase 3：迁移 L5

1. `git mv` L5 文件。
2. 更新 README、toc、changelog、知识文档内引用。
3. 旧文件名如果已经被外部文档引用，可保留迁移说明。

### Phase 4：增加检查脚本

后续可增加：

```text
.ai/L0#Execution/workflows/ai-memory-naming-check.md
```

用于检查：

1. 非 README 文件是否小写。
2. 是否存在下划线。
3. L3 是否匹配编号格式。
4. L4 是否匹配日期格式。
5. L5 是否使用领域前缀和文档类型后缀。

---

## 八、当前策略

当前不立即批量重命名旧文件。

原因：

1. `.ai` 内部已有大量交叉引用。
2. L4 操作日志属于历史记录，不应随意改写历史路径。
3. L5 文档已经被本轮测试、SFT、NewAPI 分析引用。
4. 直接重命名会影响后续 AI 检索上下文。

正确做法：

```text
先固化命名规范 -> 再按迁移清单逐批 git mv -> 每批更新引用 -> 每批记录 changelog
```
