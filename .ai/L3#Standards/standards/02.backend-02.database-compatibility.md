# 三库兼容开发规范

> **级别**：零容忍
> **创建日期**：2026-05-13

---

## 规则

所有数据库代码**必须**同时兼容 SQLite、MySQL >= 5.7.8、PostgreSQL >= 9.6。

---

## 优先使用 GORM 抽象

```go
// ✅ 优先用 GORM 方法
db.Create(&record)
db.Where("status = ?", 1).Find(&records)
db.Model(&record).Updates(map[string]interface{}{...})
```

---

## Raw SQL 注意事项

| 差异点 | MySQL/SQLite | PostgreSQL |
|--------|-------------|-----------|
| 列引用 | `` `column` `` | `"column"` |
| 布尔值 | `1` / `0` | `true` / `false` |
| 保留字 | 需要反引号 | 需要双引号 |

**使用项目变量：**
- `commonGroupCol` / `commonKeyCol` — 保留字列名
- `commonTrueVal` / `commonFalseVal` — 布尔值
- `common.UsingPostgreSQL` / `common.UsingSQLite` / `common.UsingMySQL` — DB 类型判断

---

## 禁止（无跨库 fallback）

- MySQL-only: `GROUP_CONCAT` (PostgreSQL 用 `STRING_AGG`)
- PostgreSQL-only: `@>`, `?`, JSONB 操作符
- SQLite 不支持: `ALTER COLUMN`
- DB 特定类型: 用 `TEXT` 代替 `JSONB`

---

## 迁移

- 所有 migration 必须三库可执行
- SQLite 用 `ALTER TABLE ... ADD COLUMN`（不支持 ALTER COLUMN）
- 参考 `model/main.go` 中的模式

---

## NACP 老库日志兼容边界

NACP 数据库演进允许通过“补字段”启用新能力，但必须满足以下边界：

1. **老数据不强行改语义**：NewAPI 老日志仍按原始 `logs.type` 和原始字段展示，不强行转换成 SFT 链路日志。
2. **补字段必须有安全默认值**：新增日志链路字段必须允许老日志保持空值或 `0`，例如 `trace_id=''`、`trace_seq=0`、`summary_log_id=0`、`trace_version=0`。
3. **页面必须可读**：老库执行字段补充后，`/console/log` 必须能正常加载老日志，不报错、不空白、不误展开。
4. **新旧日志必须可共存**：老日志按单行独立日志显示；新产生的 NACP SFT 日志才使用 `20/50` 摘要和 `21/29/51/52/59` 链路明细。
5. **统计不能被污染**：补字段不能改变老日志的计费、计量、统计口径；新链路日志的探测、拦截、最终错误也不能污染用户消费统计。
6. **双库是目标形态**：业务库承载用户、渠道、令牌、配置、计费等业务表；日志库承载 `logs`。迁移 SQL 必须明确区分 `SQL_DSN` 业务库和 `LOG_SQL_DSN` 日志库。

数据库升级的合格标准不是“字段加成功”，而是：

```text
旧 NewAPI 双库数据 + NACP 新字段
-> 后端启动正常
-> /api/log/grouped 正常返回老日志
-> /console/log 正常显示老日志
-> 新请求可以写入并展示 NACP SFT 链路日志
```
