## 一、设计原则

1. **不做开关操作** — 健康状态机替代简单开/关
2. **优先级和权重由人设置** — 系统只管健康状态
3. **客户无感知** — 错误在本层拦截，最多延迟稍高
4. **用户请求驱动为主** — 周期性探测只是辅助
5. **探测用最小请求** — 不用用户原始请求

## 二、健康状态存储

### 方案：channels 表加字段 + 内存缓存

**数据库（持久化）：** 在 `channels` 表新增字段：

```sql
ALTER TABLE channels ADD COLUMN health_status VARCHAR(16) DEFAULT 'healthy';
ALTER TABLE channels ADD COLUMN health_updated_at BIGINT DEFAULT 0;
ALTER TABLE channels ADD COLUMN health_fail_count INT DEFAULT 0;
ALTER TABLE channels ADD COLUMN health_success_count INT DEFAULT 0;
```

**内存缓存（高性能读取）：** 复用现有 `channelsIDM` 缓存机制，渠道选择时直接读内存。

**写入策略：** 状态变更时 → 更新内存缓存 + 异步写库。

## 四、同优先级内换渠道

### 现有逻辑

```
retry=0 → 最高优先级，随机选一个
retry=1 → 第二优先级，随机选一个
retry=2 → 第三优先级，随机选一个
```

### 改为

```
retry=0 → 最高优先级，随机选渠道 A
retry=1 → 最高优先级，排除 A，随机选渠道 B
retry=2 → 最高优先级，排除 A+B，随机选渠道 C
...直到同优先级用完...
retry=N → 降到第二优先级
```

**实现方式：** 在 context 中记录已尝试的 channel_id 列表（现有 `use_channel` 已经在做），`GetRandomSatisfiedChannel` 排除这些 ID。

## 五、探测函数（新写，轻量版）

### 为什么不用现有 testChannel

现有 `testChannel` 太重：
- 构造完整 gin context + httptest.Recorder
- 走完整 relay 流程（adaptor、billing 初始化等）
- 一次 1-5 秒

### 新的 probeChannel

```go
func ProbeChannel(channel *model.Channel, modelName string) (ok bool, latencyMs int64) {
    // 1. 构造最小请求体
    //    {"messages":[{"role":"user","content":"hi"}],"max_tokens":1,"stream":false}
    // 2. 直接 HTTP POST 到上游（用 channel 的 base_url + key）
    // 3. 超时 3 秒
    // 4. 只看 HTTP 状态码是否 2xx
    // 5. 返回结果
}
```

特点：
- 不走 relay 流程，直接 HTTP 调用
- 最小 token 消耗（max_tokens=1）
- 3 秒超时
- 只验证连通性，不处理响应内容

## 六、状态机阈值（默认值）

| 参数 | 含义 | 默认值 |
|------|------|--------|
| `probe_fail_threshold` | 连续探测失败 N 次 → Degraded | 2 |
| `degraded_probe_interval` | Degraded 下恢复探测间隔 | 5 分钟 |
| `recovery_success_threshold` | 连续成功 N 次 → Recovering | 3 |
| `recovery_observation_period` | Recovering 持续无错误时长 → Healthy | 10 分钟 |
| `recovery_fail_tolerance` | Recovering 期间出错几次回退 | 1 次即回退 |
| `probe_timeout` | 单次探测超时 | 3 秒 |
| `max_retry_same_priority` | 同优先级内最多尝试几个渠道 | 3 |

初期硬编码，后续可做成管理后台配置。

## 七、重试流程（改后）

```
用户请求进入 Relay()
    ↓
for retry loop:
    ↓
    getChannel() — 过滤 Degraded，同优先级内排除已试过的
    ↓
    relayHandler() — 转发上游
    ↓
    ├── 成功 → return
    └── 失败 → processChannelError()
            ↓
        触发 probeChannel()（异步，2次快速探测）
            ├── 探测失败 → 渠道 health_status = degraded
            └── 探测成功 → 可能偶发，不改状态
        ↓
        shouldRetry() — 判断是否继续
            ├── 同优先级还有其他渠道 → 继续循环
            └── 同优先级用完 → 降级到低优先级
```

## 八、400 错误的特殊处理

数据显示 400 错误占 75%。400 通常是参数/格式问题：
- **不应该触发渠道降级** — 不是渠道的问题，是请求格式不兼容
- **但应该切换渠道重试** — 不同渠道（不同供应商）对参数的容忍度不同
- **不应该触发探测** — 探测用最小请求，肯定能通过，没意义

**规则：** 400 错误 → 切换渠道重试，但不改变渠道健康状态。
