# NACP API 驱动测试夹具规范

> **版本**：v1.0.0
> **创建日期**：2026-05-16
> **适用范围**：本地 `localhost` 测试、`nacp.m.srl` 测试站、NACP SFT/日志/计费/渠道测试

---

## 一、定位

NACP 测试必须尽量模拟真实系统行为，而不是直接写数据库伪造结果。

标准做法是通过 API 构造测试夹具：

```text
管理员 API 调整测试渠道/分组/优先级/失效渠道
-> 普通用户 API 注册/登录
-> 普通用户 API 创建 token 并选择分组/模型限制
-> Relay API 发起真实请求
-> 管理员 API 查询日志、trace、计费、统计
-> API 恢复或清理测试夹具
```

测试可以自动化，不要求通过浏览器模拟点击；但必须尽量走与真实用户一致的 HTTP API 路径。

---

## 二、环境约定

| 项 | 约定 |
|----|------|
| 本地前端入口 | `http://localhost:23901/` |
| 本地 dashboard/admin/user API | `http://localhost:23901/api/*` |
| 本地 relay API | 从 `/api/status.server_address` 读取，通常是 `http://localhost:23900/v1/*` |
| 本地开发环境状态 | 默认由用户提前打开前后端；测试只做探测和使用，不主动启动、重启或切换端口。用户明确要求重启时，使用 `./gogogo.sh 0 restart` |
| 测试站 | `https://nacp.m.srl/` |
| 本地数据库 | 不单独维护本地数据库 |
| 测试数据库 | 本地开发和 `nacp.m.srl` 测试站共用测试站 MySQL |
| 数据库直连 | 非默认路径；只有 API 无法证明时才补充 |

本地环境探测规则：

1. 默认认为本地前端和后端已经由用户打开。
2. 测试开始先调用 `http://localhost:23901/api/status`，确认 API proxy 和 `server_address`。
3. 如果 `23901` 或 `server_address` 不可达，只报告环境不可达和探测结果，不擅自启动、重启、kill 进程或改端口。
4. 只有用户明确要求“启动本地环境”“重启后端/前端”时，才按 `gogogo.sh` 或项目脚本处理。
5. 本地 Stage A 默认连接 `nacp.m.srl` 测试站数据库；如果本地代码修改涉及数据库结构，允许直接迁移/修改测试站数据库。
6. 如果本地代码已迁移测试站数据库导致线上测试站暂时不可用，应继续完成本地验证，再部署新程序镜像到测试站恢复结构/代码一致性。
7. SFT 容错重试测试前必须确认本地测试库 option `RetryTimes >= 1`；建议设为 `2`，这样才能覆盖 A 失败、B 成功以及全部失败收尾。`RetryTimes=0` 只适合作为“无跨优先级重试”兼容场景单测。

本地开发固定端口：

| 服务 | 固定端口 | 启动方式 |
|------|----------|----------|
| 前端 Vite | `23901` | `./gogogo.sh 0 b` 或 `./gogogo.sh 0 c` |
| 后端 Go | `23900` | `./gogogo.sh 0 a` 或 `./gogogo.sh 0 c` |
| 测试数据库 | `nacp.m.srl` 测试站 MySQL | `gogogo.sh` 启动时从测试站 `.env` 读取 `SQL_DSN` |

本地环境重启命令：

```bash
./gogogo.sh 0 restart
```

该命令会直接使用线上测试数据库，并强制导出：

```text
PORT=23900
```

`SQL_DSN` 不写入文档；由脚本从 `nacp.m.srl:/opt/nacp/.env` 读取并替换为本地可访问的测试站数据库地址。

脚本启动前只清理 `23900`、`23901` 的端口占用，不再启动或维护本地 MySQL。

## 三、测试夹具类型

### 3.1 用户夹具

通过 API 创建普通用户：

```text
POST /api/user/register
POST /api/user/login
```

命名建议：

```text
nacp_t_<功能>_<YYYYMMDDHHmmss>
```

要求：

1. 必须是普通用户，不使用管理员 token 直接 relay。
2. token 必须由该普通用户自己创建。
3. 真实 relay 请求必须能在管理员日志里回查到 username/token_name/request_id。

### 3.2 Token 夹具

通过普通用户 API 创建 token：

```text
POST /api/token/
POST /api/token/:id/key
```

要求：

1. token 名称带测试场景和时间戳。
2. 明确分组，例如 `default`、`nacp-sft-test-*`。
3. 必要时开启模型限制，确保请求只走目标模型。
4. 不在 `.ai` 文档中记录真实 token key。

### 3.3 渠道夹具

通过管理员 API 创建或更新渠道：

```text
POST /api/channel/
PUT /api/channel/
DELETE /api/channel/:id
GET /api/channel/test/:id
```

典型场景：

1. **有效渠道**：使用真实测试渠道 key，验证最终成功。
2. **失效渠道**：使用无效 key、错误 base_url、mock 500/429 base_url，验证 51/52。
3. **探测渠道**：放在同分组后续优先级，验证 29/59。
4. **优先级链路**：同分组内配置 A/B/C/D 优先级，验证 A 失败后同渠道重试、并行探测、切换到健康渠道。

命名建议：

```text
NACP-fixture-<scenario>-<stamp>-A
NACP-fixture-<scenario>-<stamp>-B
NACP-fixture-<scenario>-<stamp>-C
```

### 3.4 分组与优先级夹具

通过渠道字段配置：

| 字段 | 用途 |
|------|------|
| `group` | 控制普通用户 token 请求可见分组 |
| `priority` | 控制同组渠道顺位 |
| `weight` | 控制同优先级权重 |
| `models` | 限定目标模型 |
| `test_model` | 渠道测试/探测模型 |
| `status` | 启用/禁用 |
| `health_status` | NACP 健康状态；测试时优先通过行为触发，不手改为主 |

要求：

1. 测试分组必须独立命名，不污染 `default`，除非测试目标就是 default。
2. 测试前记录原始渠道设置。
3. 测试后通过 API 恢复或删除测试夹具。

## 四、SFT 容错重试真实测试矩阵

| 场景 | 渠道布局 | 预期用户结果 | 预期日志 |
|------|----------|--------------|----------|
| 直接成功 | A 有效 | 成功 | `2`，不需要展开 |
| A 失败 B 成功 | A 失效，B 有效 | 成功 | 摘要 `20`，展开含 `51 -> 29/59 -> 21` |
| 全部失败 | A/B/C 均失效 | 失败 | 摘要 `50`，展开以 `52` 收尾 |
| 探测成功但正式失败 | A 失效，B 探测成功但 B 正式请求失败，C 有效 | 最终成功 | B 正式失败必须进入新的重试段，最终 `21` |
| 探测超时 | A 失败，B/C 探测未及时返回 | 策略按已知健康和顺位决策 | trace 顺序可解释，不能乱序或重复 |
| 客户端取消 | 用户请求中断 | 客户端取消 | `499`/`52` 或可解释失败，不得错误扣费 |

## 五、验收字段

每个真实测试必须至少记录：

| 层 | 必查字段 |
|----|----------|
| 用户响应 | HTTP status、body、request_id |
| `/api/log/grouped` | `id/type/trace_role/request_id/trace_id/trace_seq/channel/quota/prompt_tokens/completion_tokens` |
| `/api/log/trace` | 步骤顺序、20/50 展开来源、29/59 token/quota、21/52 终态 |
| 计费 | 用户余额、token 余额、used_quota、request_count |
| 统计 | user/channel/model/log stat 不被 51/52/29/59 污染 |
| 前端 | `/console/log` 能看到摘要行并展开字段完整 |

## 六、老库日志兼容验收

当本次变更涉及 `logs` 表字段、日志 API、日志页面或业务库/日志库分离时，必须额外执行老库兼容验收。

验收边界：

1. 老库可以通过 SQL 补充 NACP 新字段，但不迁移老日志语义。
2. 老日志的 `trace_version` 默认为 `0`，链路字段为空或 `0`。
3. `/api/log/grouped` 对老日志必须返回正常列表，不因缺少链路数据报错。
4. `/console/log` 必须正常显示老日志，不空白、不把老日志误判成 `20/50` 链路。
5. 新产生的 NACP SFT 日志必须能和老日志混合分页、筛选、排序。
6. 统计、计费、计量只按设计口径读取，不因为补字段改变老日志历史数据。

最低测试路径：

```text
准备 NewAPI 老双库样本
-> 对业务库补 channels 健康字段
-> 对日志库补 logs 链路字段
-> 启动 NACP
-> API 查询老日志
-> 前端打开 /console/log
-> 发起一条新 SFT 请求
-> 验证新旧日志混合显示和统计口径
```

---

## 七、禁止项

1. 禁止把手动 DB seed 当成完整真实链路测试结论。
2. 禁止在 `.ai` 中记录真实用户 token key、渠道 key、管理员密码。
3. 禁止测试后遗留大量启用的失效渠道影响后续测试。
4. 禁止直接修改正式运营站渠道来做破坏性测试。
5. 禁止只看截图不查 API 字段。

DB seed 只允许用于：

1. 前端纯展示开发。
2. 兼容旧日志展示。
3. API 不可达时的临时 UI fixture。

DB seed 结果必须明确标注“非真实链路”。
