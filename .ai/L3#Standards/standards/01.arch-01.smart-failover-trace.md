# NACP 智能容错链路（SFT）架构方案

> **状态**：已实现，持续演进
> **优先级**：P0
> **创建日期**：2026-05-13
> **更新日期**：2026-05-17

---

## 一、设计原则

1. **不做开关操作** — 健康状态机替代简单开/关
2. **优先级和权重由人设置** — 系统只管健康状态
3. **客户无感知** — 错误在本层拦截，最多延迟稍高
4. **用户请求驱动** — 用真实用户请求按渠道顺位验证可用性
5. **不做链路内轻量探测** — 不用额外最小 token probe 预判渠道

---

## 二、健康状态存储

### 方案：channels 表加字段 + 内存缓存

**数据库（持久化）：** 在 `channels` 表新增字段：

```sql
ALTER TABLE channels ADD COLUMN health_status VARCHAR(16) DEFAULT 'healthy';
ALTER TABLE channels ADD COLUMN health_updated_at BIGINT DEFAULT 0;
ALTER TABLE channels ADD COLUMN health_fail_count INT DEFAULT 0;
ALTER TABLE channels ADD COLUMN health_success_count INT DEFAULT 0;
```

**内存缓存（高性能读取）：** 复用现有 `channelsIDM` 缓存机制，渠道选择时直接读内存。

**写入策略：** 状态变更时 → 更新内存缓存 + 异步写库。

## 三、代码改动方案

### 改动方式：在现有逻辑基础上增强，不推翻重来

| 文件 | 改动内容 |
|------|---------|
| `model/channel.go` | 新增健康状态字段 |
| `model/channel_cache.go` → `GetNextSatisfiedChannel` | SFT 顺位选择：优先级降序、权重降序、ID 升序兜底 |
| `middleware/distributor.go` → `Distribute` | 第一跳渠道选择也使用 SFT 顺位选择器，不能走随机/健康状态跳过 |
| `controller/relay.go` → `Relay` 循环 | 后续顺位正式请求重试，不做链路内轻量探测 |
| `controller/relay.go` → `processChannelError` | 触发状态机转换 |
| `service/channel_health.go` | 健康状态机，不在启动时拉起后台轻量探测 |
| `service/channel_probe.go` | 历史/专项测试能力保留，默认不参与用户请求链路 |

### 不改动的部分

- 优先级/权重的设置和含义不变
- 分组机制不变
- Channel Affinity 不变
- 管理后台的渠道 CRUD 不变

## 四、顺位渠道选择

### 原逻辑问题

```
retry=0 → 最高优先级，随机选一个
retry=1 → 第二优先级，随机选一个
retry=2 → 第三优先级，随机选一个
```

这会导致同优先级下没有穷尽，也会让 `RetryTimes` 和优先级数量绑定。

### 当前逻辑

```
尝试集合 = 同分组 + 同模型 + enabled + 本链路尚未尝试
排序 = priority DESC, weight DESC, channel_id ASC

示例：
P100/W90 -> P100/W60 -> P100/W10 -> P90/W80 -> P90/W20
```

同优先级下不再随机抽取；权重在 SFT 链路里表示固定尝试顺序。权重相同用 `channel_id ASC` 做稳定兜底，保证日志链路可复现。

## 五、轻量探测的当前边界

NACP 当前主流程不再使用轻量探测作为容错重试的一部分。

保留 `service/channel_probe.go` 的原因：

1. 兼容已经存在的 `29/59` 历史日志展示。
2. 保留专项诊断或后续独立健康检查能力。
3. 不作为用户请求链路中的选择依据。

标准容错链路中不应生成新的 `29/59`。如果将来重新启用探测，必须重新进入 CAF 评审，明确费用归属、日志语义、排序、统计隔离和超时策略。

## 六、状态机阈值（默认值）

| 参数 | 含义 | 默认值 |
|------|------|--------|
| `probe_fail_threshold` | 历史探测阈值，当前默认主流程不使用 | 2 |
| `degraded_probe_interval` | 历史恢复探测间隔，当前启动不拉起后台探测 | 5 分钟 |
| `recovery_success_threshold` | 历史探测恢复阈值，当前默认主流程不使用 | 3 |
| `recovery_observation_period` | Recovering 持续无错误时长 → Healthy | 10 分钟 |
| `recovery_fail_tolerance` | Recovering 期间出错几次回退 | 1 次即回退 |
| `probe_timeout` | 历史单次探测超时，当前默认主流程不使用 | 3 秒 |
| `max_retry_same_priority` | 历史参数，当前 SFT 顺位链路不使用 | 3 |

初期硬编码，后续可做成管理后台配置。

## 七、重试流程（当前标准）

```
用户请求进入 Relay()
    ↓
for retry loop:
    ↓
    getChannel() — 按分组/模型/优先级/权重选择下一个未尝试的启用渠道
    ↓
    relayHandler() — 用用户原始请求转发上游
    ↓
    ├── 成功 → return
    └── 失败 → processChannelError() 记录 51 拦截错误
        ↓
        shouldRetry() — 判断错误类型是否允许继续
            ├── 还有可尝试渠道 → 继续正式请求下一个渠道
            └── 分组内渠道耗尽或不可重试 → 记录 52，并由 summary 生成 50
```

核心规则：

0. 第一跳和后续跳必须共用同一套顺位规则；不能第一跳随机、后续跳顺位，否则会出现 `43 → 42` 这类不符合优先级/权重定义的链路。
1. A1 失败后，不向用户暴露 A1 错误，记录 `51`。
2. 直接选择下一个正式渠道 B，发送真实用户请求 B1。
3. B1 失败后继续记录 `51`，再尝试 C。
4. 任一渠道成功后，最终消费日志为 `21`，摘要为 `20`。
5. 所有可尝试渠道失败后，最终用户可见错误为 `52`，摘要为 `50`。
6. 当前标准链路不生成 `29/59`。
7. `RetryTimes` 不作为顺位链路的硬渠道数量上限；链路应以“同分组内没有下一个可尝试渠道”为耗尽条件。
8. 顺位链路不因历史 `health_status=degraded` 提前跳过渠道；只要渠道启用且匹配分组/模型，就应由真实用户请求验证一次。
9. 顺位链路必须穷尽同分组同模型下所有开启渠道；例如 10 个渠道里 3 个关闭，则最终 `52/50` 前应出现 7 次正式尝试。用户取消、总超时、指定单渠道、Channel Affinity 明确禁止继续、系统级 skip 错误除外。
10. 同优先级渠道按权重从大到小尝试，不做随机；权重相同按渠道 ID 升序稳定排序。
11. 日志列表的渠道链路超过 2 个时默认只显示首尾，例如 `33 → ... → 36`；完整链路必须能通过 tooltip 和展开行查看。

### 时间小计展示

展开行只显示本行日志自己的时间信息，不显示链路累计。链路总耗时只在 `20/50` 摘要行体现。

时间点定义：

| 字段 | 含义 |
|------|------|
| `downstream_received_at_ms` | 下游↑：NACP 收到用户请求 |
| `upstream_started_at_ms` | 上游↑：NACP 向当前渠道发出正式请求 |
| `upstream_finished_at_ms` | 上游↓：NACP 收到当前渠道成功或错误结果 |
| `downstream_finished_at_ms` | 下游↓：NACP 将最终成功或失败结果返回给用户 |

展示规则：

1. 链路第一行：显示 `下游↑`、`上游↑`、`上游↓`。
2. 中间行：只显示 `上游↑`、`上游↓`。
3. 最后一行（`21` 或 `52`）：显示 `上游↑`、`上游↓`、`下游↓`。
4. 直接成功 `2`：同时是第一行和最后一行，显示 `下游↑`、`上游↑`、`上游↓`、`下游↓`。
5. 老日志没有 `other.timing` 时，前端只能用 `created_at/use_time` 估算 `上游↑/上游↓`，不能强行补写数据库。

## 八、400 错误的特殊处理

数据显示 400 错误占 75%。400 通常是参数/格式问题：
- **不应该触发渠道降级** — 不是渠道的问题，是请求格式不兼容
- **但应该切换渠道重试** — 不同渠道（不同供应商）对参数的容忍度不同
- **不应该触发探测** — 探测用最小请求，肯定能通过，没意义

**规则：** 400 错误 → 切换渠道重试，但不改变渠道健康状态。
