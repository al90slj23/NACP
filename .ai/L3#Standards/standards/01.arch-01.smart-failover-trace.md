# NACP 智能容错链路（SFT）架构方案

> **状态**：已实现，持续演进
> **优先级**：P0
> **创建日期**：2026-05-13
> **更新日期**：2026-05-17

---

## 一、设计原则

1. **不做开关操作** — 健康状态机替代简单开/关
2. **优先级和权重由人设置** — 系统只管健康状态
3. **客户无感知** — 错误在本层拦截，最多延迟稍高
4. **用户请求驱动** — 用真实用户请求按渠道顺位验证可用性
5. **不做链路内轻量探测** — 不用额外最小 token probe 预判渠道

---

## 二、健康状态存储

### 方案：channels 表加字段 + 内存缓存

**数据库（持久化）：** 在 `channels` 表新增字段：

```sql
ALTER TABLE channels ADD COLUMN health_status VARCHAR(16) DEFAULT 'healthy';
ALTER TABLE channels ADD COLUMN health_updated_at BIGINT DEFAULT 0;
ALTER TABLE channels ADD COLUMN health_fail_count INT DEFAULT 0;
ALTER TABLE channels ADD COLUMN health_success_count INT DEFAULT 0;
```

**内存缓存（高性能读取）：** 复用现有 `channelsIDM` 缓存机制，渠道选择时直接读内存。

**写入策略：** 状态变更时 → 更新内存缓存 + 异步写库。

## 三、代码改动方案

### 改动方式：在现有逻辑基础上增强，不推翻重来

| 文件 | 改动内容 |
|------|---------|
| `model/channel.go` | 新增健康状态字段 |

- 优先级/权重的设置和含义不变
- 分组机制不变
- Channel Affinity 不变
- 管理后台的渠道 CRUD 不变

## 四、顺位渠道选择

### 原逻辑问题

```
retry=0 → 最高优先级，随机选一个
retry=1 → 第二优先级，随机选一个
retry=2 → 第三优先级，随机选一个
```

这会导致同优先级下没有穷尽，也会让 `RetryTimes` 和优先级数量绑定。

### 当前逻辑

```
尝试集合 = 同分组 + 同模型 + enabled + 本链路尚未尝试
排序 = priority DESC, weight DESC, channel_id ASC

示例：
P100/W90 -> P100/W60 -> P100/W10 -> P90/W80 -> P90/W20
```

同优先级下不再随机抽取；权重在 SFT 链路里表示固定尝试顺序。权重相同用 `channel_id ASC` 做稳定兜底，保证日志链路可复现。

## 五、轻量探测的当前边界

NACP 当前主流程不再使用轻量探测作为容错重试的一部分。

保留 `service/channel_probe.go` 的原因：

1. 兼容已经存在的 `29/59` 历史日志展示。
2. 保留专项诊断或后续独立健康检查能力。
3. 不作为用户请求链路中的选择依据。

标准容错链路中不应生成新的 `29/59`。如果将来重新启用探测，必须重新进入 CAF 评审，明确费用归属、日志语义、排序、统计隔离和超时策略。

## 六、状态机阈值（默认值）

| 参数 | 含义 | 默认值 |
|------|------|--------|
| `probe_fail_threshold` | 历史探测阈值，当前默认主流程不使用 | 2 |
| `degraded_probe_interval` | 历史恢复探测间隔，当前启动不拉起后台探测 | 5 分钟 |
| `recovery_success_threshold` | 历史探测恢复阈值，当前默认主流程不使用 | 3 |
| `recovery_observation_period` | Recovering 持续无错误时长 → Healthy | 10 分钟 |
| `recovery_fail_tolerance` | Recovering 期间出错几次回退 | 1 次即回退 |
| `probe_timeout` | 历史单次探测超时，当前默认主流程不使用 | 3 秒 |
| `first_byte_timeout` | 单渠道正式请求等待上游首字/响应头的上限，不限制流式完整输出 | 20 秒 |
| `total_retry_timeout` | 从 NACP 接收到 relay 请求开始计算的 SFT 总首字预算 | 60 秒 |
| `max_retry_same_priority` | 历史参数，当前 SFT 顺位链路不使用 | 3 |

初期硬编码，后续可做成管理后台配置。

## 七、重试流程（当前标准）

```
用户请求进入 Relay()
    ↓
for retry loop:
    ↓
    getChannel() — 按分组/模型/优先级/权重选择下一个未尝试的启用渠道
    ↓
    relayHandler() — 用用户原始请求转发上游
    ↓
    ├── 成功 → return
    └── 失败 → processChannelError() 记录 51 拦截错误
        ↓
        shouldContinueSequentialFailover() — 判断错误类型是否允许继续
            ├── 还有可尝试渠道 → 继续正式请求下一个渠道
            └── 分组内渠道耗尽或不可重试 → 记录 52，并由 summary 生成 50
```

核心规则：

0. 第一跳和后续跳必须共用同一套顺位规则；不能第一跳随机、后续跳顺位，否则会出现 `43 → 42` 这类不符合优先级/权重定义的链路。
1. A1 失败后，不向用户暴露 A1 错误，记录 `51`。
2. 直接选择下一个正式渠道 B，发送真实用户请求 B1。
3. B1 失败后继续记录 `51`，再尝试 C。
4. 任一渠道成功后，最终消费日志为 `21`，摘要为 `20`。
5. 所有可尝试渠道失败后，最终用户可见错误为 `52`，摘要为 `50`。
6. 当前标准链路不生成 `29/59`。
7. `RetryTimes` 不作为顺位链路的硬渠道数量上限；链路应以“同分组内没有下一个可尝试渠道”为耗尽条件。
8. 顺位链路不因历史 `health_status=degraded` 提前跳过渠道；只要渠道启用且匹配分组/模型，就应由真实用户请求验证一次。
9. 顺位链路必须穷尽同分组同模型下所有开启渠道；例如 10 个渠道里 3 个关闭，则最终 `52/50` 前应出现 7 次正式尝试。用户取消、总超时、指定单渠道、Channel Affinity 明确禁止继续、系统级 skip 错误除外。
10. 同优先级渠道按权重从大到小尝试，不做随机；权重相同按渠道 ID 升序稳定排序。
11. 日志列表的渠道链路超过 2 个时默认只显示首尾，例如 `33 → ... → 36`；完整链路必须能通过 tooltip 和展开行查看。
12. SFT 总时间不是“启动下一跳前的软检查”，而是首字硬预算：从 `relay_received_at` 开始最多 60 秒。每次正式上游请求的首字等待取 `min(20 秒, 总剩余时间)`。
13. 首字限制只覆盖上游响应头/首字到达前的等待；一旦流式响应已经开始，完整流式输出不受 `first_byte_timeout` 和 `total_retry_timeout` 限制，继续由原有流式扫描/保活/流式超时机制处理。
14. `RELAY_TIMEOUT` 仍保留为 NewAPI 原生 HTTP client 全局生命周期超时，默认 `0` 表示关闭。它不是 SFT 业务超时，不能用它替代首字限制；设置过短会截断长流式输出。

### 进入与退出重试判定

当前 SFT 主流程不再以 NewAPI 原生 `AutomaticRetryStatusCodeRanges` 作为第一判断，也不以 `RetryTimes` 作为渠道数量上限。主判断函数是 `controller/relay.go` 的 `shouldContinueSequentialFailover()`。

进入 SFT 继续顺位重试的基本条件：

1. 已经完成某个上游正式请求尝试。
2. 本次尝试产生 `NewAPIError`。
3. 错误状态码不是 `2xx`。
4. 没有命中下面的不重试短路条件。

当前会阻止继续重试的短路条件：

| 条件 | 说明 |
|------|------|
| `openaiErr == nil` | 请求已经成功，不再重试 |
| `2xx` 状态码 | 视为成功，不再重试 |
| `ShouldSkipRetryAfterChannelAffinityFailure(c) == true` | 渠道亲和规则明确要求亲和渠道失败后不继续 |
| `specific_channel_id` 存在 | 用户或系统明确指定单一渠道，不切换其他渠道 |
| `IsSkipRetryError(err) && !IsChannelError(err)` | 非渠道类错误显式标记 `skipRetry` |
| `NewRequestCanceledErrorIfDone(c) != nil` | 用户取消或下游连接关闭，生成 499，不继续消耗上游 |
| `SFTRetryTimedOut(c) == true` | SFT 总首字预算耗尽 |
| `getChannel()` 无法返回下一个渠道 | 同分组同模型下没有可继续尝试的启用渠道，或选渠失败 |
| 保护性循环上限 | 当前主循环有 100 次保护上限，防止异常配置导致死循环 |

`skipRetry` 的实际语义：

1. `skipRetry` 不是绝对不重试。
2. 如果错误码是 `channel:*`，当前 SFT 仍允许继续顺位重试。
3. 如果错误码不是 `channel:*`，并且带 `skipRetry`，则停止重试。

常见不进入 SFT 的错误类别：

| 类别 | 例子 | 原因 |
|------|------|------|
| 本地请求体错误 | 请求体读取失败、请求体过大、JSON 格式错误 | 请求未形成有效上游尝试 |
| 本地请求校验错误 | `invalid_request` | 用户请求本身不合法 |
| 请求转换错误 | `convert_request_failed` | NACP 本地转换失败，不应换渠道掩盖 |
| API 类型错误 | `invalid_api_type` | 本地路由/适配类型错误 |
| 本地序列化错误 | `json_marshal_failed` | NACP 内部处理失败 |
| 模型价格错误 | `model_price_error` | 本地计费配置缺失或异常 |
| 用户额度不足 | `insufficient_user_quota`、`pre_consume_token_quota_failed` | 用户账务问题，不是渠道问题 |
| 本地计费/数据错误 | `query_data_error`、`update_data_error` | 本地数据处理失败 |
| 用户取消 | 499 `client request closed` | 下游已经断开，不继续请求上游 |
| 违规/安全拦截 | `violation_fee.grok.csam` 等 | 本地安全策略，不应通过切渠道绕过 |

常见会进入 SFT 的错误类别：

| 类别 | 例子 | 说明 |
|------|------|------|
| 上游 400 | 参数或格式不兼容 | 会切换渠道重试，但不改变渠道健康状态 |
| 上游 401/403 | 认证、权限、模型权限问题 | 会重试；401 仍可能触发自动禁用规则 |
| 上游 404/409/422 | 模型不存在、冲突、参数不支持 | 会重试，因为其他供应商可能兼容 |
| 上游 408/504/524 | 请求超时、网关超时 | 会重试；健康状态有特殊处理 |
| 上游 429 | 限流或额度类上游错误 | 会重试 |
| 上游 5xx | 500、502、503、505-599 | 会重试，并进入渠道健康状态统计 |
| 网络错误 | 连接失败、`do_request_failed` | 会重试 |
| 首字超时 | `channel:response_time_exceeded` | 会重试，受 20 秒单次首字和 60 秒总预算约束 |
| 上游响应异常 | 空响应、坏响应、响应体解析失败 | 通常会重试，具体取决于错误是否被标记为非渠道类 `skipRetry` |
| `channel:*` 错误 | key、映射、覆盖参数、AWS client 等渠道错误 | 当前 SFT 允许继续顺位重试 |

### 时间小计展示

展开行只显示本行日志自己的时间信息，不显示链路累计。链路总耗时只在 `20/50` 摘要行体现。

时间点定义：

| 字段 | 含义 |
|------|------|
| `downstream_received_at_ms` | 下游↑：NACP 收到用户请求 |
| `upstream_started_at_ms` | 上游↑：NACP 向当前渠道发出正式请求 |
| `upstream_finished_at_ms` | 上游↓：NACP 收到当前渠道成功或错误结果 |
| `downstream_finished_at_ms` | 下游↓：NACP 将最终成功或失败结果返回给用户 |

展示规则：

1. 链路第一行：显示 `下游↑`、`上游↑`、`上游↓`。
2. 中间行：只显示 `上游↑`、`上游↓`。
3. 最后一行（`21` 或 `52`）：显示 `上游↑`、`上游↓`、`下游↓`。
4. 直接成功 `2`：同时是第一行和最后一行，显示 `下游↑`、`上游↑`、`上游↓`、`下游↓`。
5. 老日志没有 `other.timing` 时，前端只能用 `created_at/use_time` 估算 `上游↑/上游↓`，不能强行补写数据库。

## 八、400 错误的特殊处理

数据显示 400 错误占 75%。400 通常是参数/格式问题：
- **不应该触发渠道降级** — 不是渠道的问题，是请求格式不兼容
- **但应该切换渠道重试** — 不同渠道（不同供应商）对参数的容忍度不同
- **不应该触发探测** — 探测用最小请求，肯定能通过，没意义

**规则：** 400 错误 → 切换渠道重试，但不改变渠道健康状态。

400 的边界：

1. 400 不是“不可重试”。
2. 400 不应该让渠道降级，也不应该触发轻量探测。
3. 400 仍应进入顺位正式请求重试，因为不同上游供应商、不同二开平台对同一请求参数的兼容度可能不同。
4. 如果 400 是 NACP 本地请求校验或转换阶段产生，并且带有非渠道类 `skipRetry`，则不进入 SFT；只有已经进入上游正式尝试后的 400 才按上游兼容性问题处理。
