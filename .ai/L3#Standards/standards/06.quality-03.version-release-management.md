# NACP 版本与发布管理规范

> **版本**：v1.0.0
> **创建日期**：2026-05-15

---

## 一、版本记录三层模型

NACP 使用三层版本记录：

```text
VERSION       -> 当前构建版本源
CHANGELOG.md  -> 仓库内长期版本变更记录
GitHub Release -> 正式发布记录、tag、构建产物和发布说明
```

`.ai/L4#Changelog (Changelog)` 继续记录内部操作过程，不替代根目录 `CHANGELOG.md`。

---

## 二、VERSION

`VERSION` 是本地构建和 Docker 构建的版本源。

当前开发版本：

```text
classic-plus-0.2.0-dev
```

规则：

1. 开发中版本使用 `classic-plus-x.y.z-dev`。
2. 正式发布版本去掉 `-dev`。
3. `Dockerfile` 和前端构建读取 `VERSION`。
4. `VERSION` 不能为空。

---

## 三、CHANGELOG.md

根目录 `CHANGELOG.md` 记录对人可读的版本变更。

每个版本至少包含：

1. 新增功能。
2. 行为变更。
3. 数据库变更。
4. API/前端变更。
5. 兼容性说明。
6. 测试与验收摘要。
7. 已知风险。

开发中版本使用：

```markdown
## classic-plus-0.2.0-dev (Unreleased)
```

正式发布时改为：

```markdown
## classic-plus-0.2.0 (2026-xx-xx)
```

### 3.1 固定章节结构

每个版本必须使用固定结构，缺省项也要写“无”：

```markdown
## classic-plus-x.y.z (YYYY-MM-DD)

### Summary

### Added

### Changed

### Fixed

### Database

### API and Frontend

### Compatibility

### Testing

### Known Risks

### Internal References
```

说明：

1. `Summary` 写这个版本一句话定位。
2. `Added` 写新增能力。
3. `Changed` 写行为变化。
4. `Fixed` 写修复。
5. `Database` 写表、字段、索引、迁移。
6. `API and Frontend` 写接口和页面变化。
7. `Compatibility` 写和 NewAPI 原版、旧日志、旧配置、旧数据的兼容影响。
8. `Testing` 写 CAF 测试结果摘要。
9. `Known Risks` 写剩余风险。
10. `Internal References` 链接 `.ai` 内部详版、测试计划、执行报告。

### 3.2 版本跃迁详版

每次 minor 或 major 版本跃迁，必须在 `L5#Knowledge (Knowledge)` 建一份详版升级说明。

命名格式：

```text
release-classic-plus-x-y-z-upgrade-review.md
```

例如：

```text
release-classic-plus-0-2-0-upgrade-review.md
```

用途：

1. 详细说明从上一个版本升级到本版本发生了什么。
2. 记录行为差异、数据差异、兼容策略。
3. 连接 CAF 测试计划和测试报告。
4. 作为 GitHub Release 正文的主要素材来源。

---

## 四、Git tag

正式版本必须打 Git tag。

格式：

```text
classic-plus-x.y.z
```

示例：

```text
classic-plus-0.2.0
```

不使用：

```text
v0.2.0
```

原因：

1. NACP 是 NewAPI v0.13.2 classic 线的增强分支。
2. `classic-plus-*` 能避免和上游 NewAPI tag 混淆。
3. GitHub Actions 当前 tag 触发规则支持任意 tag。

---

## 五、GitHub Release

正式发布必须创建 GitHub Release。

Release 内容至少包含：

1. 版本号。
2. 对应 tag。
3. 变更摘要。
4. 数据库迁移说明。
5. 部署说明。
6. CAF 测试结论。
7. 已知风险和回滚建议。
8. 构建产物和 checksum。

GitHub Release 是对外/团队共享的正式发布记录；`.ai/L4#Changelog (Changelog)` 是内部过程审计。

---

## 六、发布前门禁

正式发布前必须满足：

1. `VERSION` 已从 `-dev` 改为正式版本。
2. `CHANGELOG.md` 已把 `Unreleased` 改为发布日期。
3. CAF 测试结论为通过或有条件通过。
4. 数据库迁移已验证。
5. Docker 构建成功。
6. Git tag 已创建。
7. GitHub Release 已创建。

---

## 七、推荐命令

```bash
printf 'classic-plus-0.2.0\n' > VERSION
git add VERSION CHANGELOG.md
git commit -m "chore(release): prepare classic-plus-0.2.0"
git tag -a classic-plus-0.2.0 -m "classic-plus-0.2.0"
git push origin main
git push origin classic-plus-0.2.0
```

发布后在 GitHub Release 中使用 tag `classic-plus-0.2.0` 创建正式 release。
