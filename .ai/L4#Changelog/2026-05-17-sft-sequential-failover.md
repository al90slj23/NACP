# 2026-05-17 — SFT 顺位正式请求容错收敛

## 操作内容

将 NACP SFT 容错重试策略从“同渠道重试 + 轻量探测预热 + 复杂健康预判”收敛为“同分组渠道顺位正式请求”：

1. A 渠道正式请求失败后，错误在 NACP 层拦截，记录 `51`。
2. 直接选择同分组下一个可尝试渠道 B，发送真实用户请求。
3. B 失败则继续 C，直到成功或同分组渠道耗尽。
4. 任一渠道成功，最终消费日志为 `21`，摘要为 `20`。
5. 全部渠道失败，最终用户可见错误为 `52`，摘要为 `50`。
6. 标准用户请求链路不再生成轻量探测 `29/59`。
7. `RetryTimes` 不再作为该链路的硬渠道数量上限，耗尽条件是同分组下没有下一个可尝试渠道。
8. 顺位容错链路不再预先跳过 `health_status=degraded` 的渠道；只要渠道启用、匹配分组和模型、并且本链路尚未尝试，就应该由真实用户请求验证一次。
9. 同优先级渠道不再按权重随机，而是按 `priority DESC, weight DESC, channel_id ASC` 稳定排序；最终失败前必须尝试完同分组同模型下所有开启渠道。
10. SFT 主链路不再复用旧 `shouldRetry` 的状态码/重试次数策略，避免配置或状态码在渠道未穷尽前提前截断；用户取消、总超时、指定单渠道、Channel Affinity 禁止继续、系统级 skip 错误除外。
11. 日志列表渠道列对长链路做首尾折叠显示，例如 `33 → ... → 36`；tooltip 和展开行保留完整链路。

## 文件变更

- 更新 `controller/relay.go`：主 Relay 循环改为简单顺位正式请求容错，不调用 `continueRetryTrace` 的探测预热路径。
- 更新 `controller/relay.go`、`service/log_info_generate.go`、`service/text_quota.go`：新日志写入本行时间小计字段 `other.timing`，包含下游↑、上游↑、上游↓、下游↓。
- 更新 `middleware/distributor.go`：第一跳渠道选择从随机选择切到同一套顺位选择器，避免第一跳与后续跳规则不一致。
- 更新 `model/channel_cache.go`、`model/ability.go`、`service/channel_select.go`：新增“下一个未尝试渠道”选择路径，按优先级/权重稳定排序并排除本链路已尝试渠道。
- 更新 `service/channel_health.go`：启动健康管理时不再拉起后台 degraded 轻量探测循环。
- 更新 `web/src/components/table/usage-logs/UsageLogsColumnDefs.jsx`：日志渠道链路超过两个时显示首尾摘要，tooltip 展示完整链。
- 更新 `web/src/components/table/usage-logs/TraceExpandRender.jsx`：展开详情中“时间记录”改为“时间小计”，每行只展示本行相关时间点，不再展示链路累计。
- 更新 `.ai/L3#Standards/standards/01.arch-01.smart-failover-trace.md`：同步 SFT 当前标准。

## 问题定位

本次调整不是强修 UI，而是针对线上测试中“分组内还有渠道但没有继续尝试”的实际链路问题做收敛：

1. 旧主循环仍受 `RetryTimes` 和 `shouldRetry` 的剩余次数影响，可能在渠道未耗尽前停止。
2. 旧渠道选择把 `retry` 映射成“优先级索引”，不是按本链路未尝试渠道集合顺位穷尽。
3. 旧选择逻辑会预先跳过 `health_status=degraded` 渠道，导致数据库里看似还有渠道，实际选择器认为不可用。
4. 旧 `excludeIDs` 分支没有“当前优先级无可选后继续找下一优先级”的简单穷尽语义。
5. 旧 `continueRetryTrace` 依赖轻量探测成功结果来决定备用渠道正式请求；探测失败、超时或未返回时，B/C 可能不会收到正式用户请求。
6. `getChannel` 返回无渠道错误时，可能覆盖最后一个真实上游错误，最终表现为 `50/52`，但链路并没有真的穷尽所有可尝试渠道。
7. 初版顺位选择仍保留了同优先级按权重随机抽取，不符合“权重大先尝试、最终失败前穷尽全部开启渠道”的链路可解释性要求。
8. 初版顺位循环仍复用旧 `shouldRetry`，可能继续受到状态码策略影响，和“穷尽开启渠道”的新定义不完全一致。
9. 本地 API 测试发现第一跳仍由 `middleware/distributor.go` 预先随机选出，后续跳才进入顺位选择，导致出现 `43 -> 42` 这类不符合 `priority DESC, weight DESC, channel_id ASC` 的链路。修复后第一跳和后续跳统一使用 `CacheGetNextSatisfiedChannel`。

## 决策记录

轻量探测带来了额外 token 成本、日志语义复杂度、排序问题、统计隔离问题和用户等待策略复杂度。当前阶段优先保证链路简单、可解释、可测试：

```text
A1 失败 -> 51
B1 失败 -> 51
C1 成功 -> 21 + 20

或：

A1 失败 -> 51
B1 失败 -> 51
C1 失败 -> 51
最终返回 -> 52 + 50
```

`29/59` 类型保留为历史兼容和专项诊断能力，不作为标准用户请求链路的一部分。

## 本地验证

测试环境：本地前端 `http://localhost:23901/`，本地后端 `http://localhost:23900/`，数据库使用 `nacp.m.srl` 测试库。

临时模型：`nacp-sft-model-044939`。临时渠道：

| 渠道 | 角色 | priority | weight |
|------|------|----------|--------|
| `42` | A | 100 | 90 |
| `43` | B | 100 | 40 |
| `44` | C | 50 | 80 |
| `45` | D | 50 | 10 |

验证结果：

1. 直接成功：A 指向本地 mock 上游，日志为 `type=2`，不生成 `20` 摘要。
2. 容错成功：A/B 失败、C 成功，trace 为 `51 -> 51 -> 21`，摘要为 `20`，`channel_path=[42,43,44]`，token/费用取成功终态。
3. 容错失败：A/B/C/D 全失败，trace 为 `51 -> 51 -> 51 -> 51 -> 52`，摘要为 `50`，`channel_path=[42,43,44,45]`。
4. 标准链路未生成新的 `29/59`，摘要中 `platform_probe_quota=0`、`probe_success_count=0`、`probe_failed_count=0`。

## 时间小计规则

展开行只展示本行自己的时间点：

1. 第一行：`下游↑`、`上游↑`、`上游↓`。
2. 中间行：`上游↑`、`上游↓`。
3. 最后一行 `21/52`：`上游↑`、`上游↓`、`下游↓`。
4. 直接成功 `2`：`下游↑`、`上游↑`、`上游↓`、`下游↓`。

链路总耗时只看 `20/50` 摘要行。旧日志没有 `other.timing` 时，前端用 `created_at/use_time` 兼容估算，不回填数据库。
