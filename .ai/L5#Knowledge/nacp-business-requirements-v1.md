# NACP 业务需求 v1 — 智能重试与渠道健康管理

> **记录日期**：2026-05-13
> **来源**：用户需求描述（第一轮+第二轮讨论）
> **状态**：需求确认阶段

---

## 一、核心设计哲学

**不做开关操作，引入健康状态机制。**

- 优先级和权重由人设置（因为有大量动态信息需要人判断）
- 系统自动管理渠道的"健康状态"，而非简单的开/关
- 只有"手动关闭"才是真正的关（完全退出所有逻辑，不再探测）
- 目标：设置好优先级和权重后，系统自动处理故障和恢复

---

## 二、渠道健康状态机

```
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  Healthy (健康)                                              │
│  ├── 正常按优先级+权重投递用户请求                              │
│  └── 触发条件：用户请求返回错误                                │
│          ↓                                                   │
│  Probing (探测中)                                            │
│  ├── 用户请求被 hold（伪延迟）                                 │
│  ├── 立即发送最小 token 探测请求验证渠道                        │
│  ├── 探测频率：高频、短周期（因为客户在等）                      │
│  ├── 探测次数：有限（控制客户等待时间）                          │
│  ├── 探测成功 → 重新发送用户原始请求到该渠道                     │
│  │       ├── 成功 → 回到 Healthy                             │
│  │       └── 又失败 → 进入 Degraded（见下方"二次失败"处理）     │
│  └── 连续探测失败 → 确认渠道有问题                             │
│          ↓                                                   │
│  Degraded (降级)                                             │
│  ├── 不再投递用户请求到该渠道                                   │
│  ├── 用户请求切换到同优先级其他渠道或降级到低优先级              │
│  ├── 周期性恢复探测（5-10分钟，辅助手段）                       │
│  └── 连续探测成功 N 次                                        │
│          ↓                                                   │
│  Recovering (恢复观察期)                                      │
│  ├── 开始接受少量用户请求（按原优先级+权重参与）                 │
│  ├── 高频监控这些请求的结果                                    │
│  ├── 出现错误 → 立即回到 Degraded                             │
│  └── 持续正常一段时间                                         │
│          ↓                                                   │
│  Healthy (健康)                                              │
│                                                              │
│  ─────────────────────────────────────────────               │
│  ManuallyDisabled (手动关闭)                                  │
│  ├── 完全退出所有逻辑，不探测，不投递                           │
│  └── 只能人工手动开启才能重新进入 Healthy                      │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 三、请求级智能重试流程

### 3.1 非流式请求（可完全重试）

```
客户请求 → 选择最高优先级渠道 → 转发上游
    ↓
上游返回错误
    ↓
本层拦截（不返回给客户）
    ↓
发送最小 token 探测请求验证该渠道
    ↓
├── 探测成功 → 可能是偶发 → 重新发送用户原始请求
│       ├── 成功 → 返回客户 ✅
│       └── 又失败 → 渠道进入 Degraded，切换其他渠道
│
└── 探测失败 → 渠道确认有问题
        ↓
    切换到同优先级其他渠道（如有）
        ├── 成功 → 返回客户 ✅
        └── 失败 → 降级到低优先级渠道
                ├── 成功 → 返回客户 ✅
                └── 所有渠道都失败 → 返回错误给客户 ❌
```

### 3.2 流式请求

**两个时间窗口：**

| 阶段 | 能否重试 | 原因 |
|------|---------|------|
| 窗口1：上游返回 HTTP 状态码之前 | ✅ 完全可以 | 还没给客户端发任何数据 |
| 窗口2：上游返回 200 后传输中途断 | ❌ 不能 | HTTP 200 已发给客户端，无法重新开始 |

**流式请求的重试策略：**
- 窗口 1（上游返回非 200）：与非流式相同，完全可以重试
- 窗口 2（传输中断）：
  1. 这次请求无法挽救，客户端会看到中断
  2. 后台立即进入研判 — 发最小 token 探测确认渠道是否真挂了
  3. 确认有问题 → 渠道进入 Degraded（恢复探测状态）
  4. 用户重新发送请求时 → 系统自动跳过该渠道，发到其他顺位渠道
  5. 研判速度很快（探测 max_tokens=1，几百毫秒），用户重发时状态大概率已更新
- 实际上大部分错误发生在窗口 1，窗口 2 的中途断流相对少见

### 3.3 "二次失败"处理

探测通过后重新发送用户请求，但又报错了：
- 该渠道立即进入 Degraded（不再给机会）
- 用户请求立即切换到其他渠道重试
- 这种情况说明渠道不稳定，需要更长的恢复观察期

## 四、探测策略

### 4.1 主动探测（基于用户请求错误触发）

- **触发**：用户请求返回错误时
- **方式**：发送最小 token 消耗的请求（如 `{"messages":[{"role":"user","content":"hi"}], "max_tokens":1}`）
- **频率**：高频、短周期（因为客户在等待）
- **次数**：有限（如 2-3 次），控制客户等待时间在可接受范围
- **目的**：快速确认渠道是否真的有问题

### 4.2 周期性探测（辅助手段）

- **触发**：渠道处于 Degraded 状态，且没有用户请求触发探测时
- **频率**：5-10 分钟
- **目的**：在无用户流量时发现渠道恢复
- **注意**：不作为主要手段，避免对上游造成 RPM 负担

### 4.3 探测不使用用户原始请求的原因

- 用户请求可能很大（长对话历史、大文件等）
- 探测目的是验证渠道连通性，不需要完整请求
- 最小请求可以更快返回结果，减少客户等待时间

## 五、告警机制

### 5.1 分组可用渠道不足告警

当某个分组内：
- 某模型的可用渠道（Healthy 状态）只剩 1-2 个时 → 告警
- 所有渠道都进入 Degraded → 紧急告警

### 5.2 渠道状态变更通知

- 渠道从 Healthy → Degraded：通知管理员
- 渠道从 Degraded → Recovering：通知管理员
- 渠道长时间处于 Degraded：定期提醒

## 六、与现有 NewAPI 机制的关系

### 保留的设计

- ✅ 优先级分层 + 权重随机（人工设置）
- ✅ 分组机制
- ✅ Auto Group 跨分组
- ✅ Channel Affinity（渠道亲和性）
- ✅ 错误日志记录

### 替换/增强的设计

| 现有 | NACP |
|------|------|
| 简单开/关状态 | 健康状态机（Healthy/Probing/Degraded/Recovering） |
| 单次错误即禁用 | 探测确认后才降级 |
| 单次测试通过即启用 | 持续验证后才恢复 |
| 全局 RetryTimes | 基于状态机的智能重试 |
| 仅跨优先级重试 | 同优先级内也切换渠道 |
| 被动等客户报告 | 基于错误日志主动触发 |
| 定时全量测试为主 | 用户请求驱动为主，定时为辅 |

### 不再需要的功能

- ❌ `AutomaticDisableChannelEnabled`（自动禁用开关）→ 被状态机替代
- ❌ `AutomaticEnableChannelEnabled`（自动启用开关）→ 被恢复机制替代
- ❌ 简单的 `ChannelStatusAutoDisabled` → 被 Degraded/Recovering 替代

## 七、实现优先级

### P0 — 请求级智能重试（客户无感知）
- 同优先级内切换渠道
- 最小 token 探测
- 伪延迟 hold 请求
- 效果：客户不再看到报错，最多延迟稍高

### P1 — 渠道健康状态机
- Healthy → Probing → Degraded → Recovering → Healthy
- 替代简单的开/关
- 基于错误触发的主动探测

### P2 — 告警与可观测性
- 分组可用渠道不足告警
- 渠道状态变更通知
- 管理后台可视化渠道健康状态
