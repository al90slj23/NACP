# NACP 单位管理与 NACP统计整合说明

> 最后更新：2026-05-18

## 背景

NACP 的“单位”来自 FlowHub 中“资源承载平台”的概念：单位本身代表一个上游平台，单位下可以挂载一个或多个账号。账号信息包括账号、密码、账户访问令牌、账户 ID。NACP 后续要基于这些配置做上游平台账号余额、令牌、模型、分组、渠道健康、用户使用和 SFT 容错链路的统一监控。

## 当前调整

1. 单位管理负责配置和查看单位、账号以及账号统计详情。
   - 单位列表操作区固定包含 `编辑`、`停用/启用`、`删除`、`详情`。
   - 点击 `详情` 会读取单位基础信息、挂载账号和账号统计源；如果挂载账号已经配置账户访问令牌和账户 ID，但尚未创建统计源，页面会自动创建统计源并立即检测。
   - 单位详情只展示自动检测和配置读取出的信息，不允许手工伪造余额、模型、令牌、充值倍率等监控值。
2. NACP统计不再作为“单位监控”的旧入口，路由固定为 `/console/nacp_stats`。
3. 旧 `/console/unit_monitor` 不保留兼容入口。
4. 单位账号统计详情在单位管理编辑弹窗内展示：
   - 没有统计源的账号，可以直接创建统计源并立即检测。
   - 已有统计源的账号，可以刷新检测结果。
   - 详情分为“摘要”和“JSON”两个视图，默认展示人类可读摘要。
   - NewAPI 兼容采集器额外读取 `/api/status`，并把平台状态保存到 `raw_json.platform_status`，用于展示系统名称、版本、额度单位、`quota_per_unit`、`price`、`stripe_unit_price`、`usd_exchange_rate`、`custom_currency_symbol`、`custom_currency_exchange_rate`、`top_up_link` 等字段。
   - `price` 当前作为“充值倍率”展示；如果上游平台没有在 `/api/status` 暴露该字段，页面必须显示 `-`，不能推断或编造。
5. NACP统计页面开始从列表页转为统计看板：
   - 顶部固定为五个主题页：`综合`、`模型&性能`、`用户&消耗`、`流水&成本`、`单位&渠道`。
   - 顶部展示统计源总数、正常/异常/待检测数量、余额汇总、采集能力、最近刷新时间。
   - 顶部展示请求质量、消耗统计、成功率、平均耗时、流式占比、用户可见错误、拦截错误、探测数量。
   - 顶部展示今日流水成本：平台收入、上游成本、预计利润、毛利率和快照覆盖度。
   - 顶部同步展示用户、渠道、模型活动、SFT 容错统计。
   - 支持 1 小时、6 小时、24 小时、7 天、30 天统计范围切换。
   - 看板内新增趋势分桶条形图，按统计范围自动选择 5 分钟、15 分钟、1 小时、6 小时或 1 天桶。
   - 下方暂保留统计源明细表，作为追溯和手动刷新入口。
   - 用户统计、模型统计、渠道健康、SFT 容错重试等板块已经接入真实聚合数据、Top 榜单和分页完整排行。
   - `单位&渠道` 页面深化为“统计源健康 + 渠道完整排行 + 分组/端点/IP 钻取 + 渠道归属”组合视图，渠道排行必须显示所属单位、单位类型、所属账号、渠道健康、优先级和权重。
   - `流水&成本` 页面深化为“今日收入 + 上游成本 + 利润 + 成本口径 + 成本排行”组合视图，明细必须显示归属单位/账号、余额/已用、快照状态、成本来源和估算状态。
6. 参考项目阅读后的看板优化原则：
   - `new-api-latest` / `one-api-latest`：保留 NewAPI 系传统的请求、额度、Token、模型趋势等基础口径，避免 NACP 看板脱离原系统的运营指标。
   - `sub2api-latest`：吸收“核心 KPI + 趋势图 + Top 排行 + 用户/模型/渠道维度钻取”的组织方式，NACP 的模型、用户、渠道页必须有完整分页排行，而不是只展示 Top 摘要。
   - `metapi-latest`：吸收站点/上游可用性、聚合快照、日志投影和成本估算的思路，NACP 的单位账号统计源需要保留快照，后续支持成本趋势和健康水位。
   - `NiceApiManager-latest`：吸收筛选维度、同步/采集状态、余额/使用额度趋势和健康状态筛选的思路，NACP 页面需要明确当前范围、数据生成时间、采集覆盖度和估算状态。
   - `ReferenceProjects` 只作为学习输入，不直接复制 UI；NACP 采用当前 Semi UI / CardPro 风格，确保和管理后台一致。

## 后端接口边界

单位管理页面复用现有 `unit_monitor` 后端能力，但查询能力增强：

- `/api/unit_monitor/?unit_id=...`
- `/api/unit_monitor/?unit_account_id=...`

这样单位编辑弹窗可以只加载当前单位下的账号统计源，而不是在前端全量拉取后推断。

单位详情按钮的读取路径：

1. `/api/unit/:id` 读取单位和挂载账号。
2. `/api/unit_monitor/?unit_id=:id&page_size=500` 读取现有统计源。
3. 对每个挂载账号：
   - 已有统计源时调用 `/api/unit_monitor/:monitor_id/check` 刷新。
   - 没有统计源且账号具备账户访问令牌和账户 ID 时，先调用 `/api/unit_monitor/` 创建统计源，再调用 `/api/unit_monitor/:monitor_id/check` 检测。
   - 缺少必要凭据时只展示配置缺失，不自动创建。
4. 检测结果中的 `raw_json.platform_status` 用于平台计价和充值信息展示。

NACP统计页面使用专用聚合接口：

- `/api/nacp_stats/overview?hours=24`
- `/api/nacp_stats/models?hours=24&p=1&page_size=10&sort=requests`
- `/api/nacp_stats/users?hours=24&p=1&page_size=10&sort=requests`
- `/api/nacp_stats/channels?hours=24&p=1&page_size=10&sort=requests`
- `/api/nacp_stats/costs?dimension=unit&p=1&page_size=10&sort=revenue`
- `/api/nacp_stats/dimensions?dimension=group&hours=24&p=1&page_size=10&sort=requests`
- `/api/nacp_stats/model_coverage?hours=24&p=1&page_size=10&sort=coverage`

这些接口只读。`overview` 用于综合态势、流水成本和趋势，排行接口用于主题页的完整分页表。当前聚合内容包括：

- 单位账号统计源：总数、启用、正常、异常、待检测、余额、已用、Token/Model/Group 采集能力、最近检测时间。
- 请求质量：当前时间范围内的用户可见请求量、成功量、错误量、成功率、平均耗时、流式占比、Token、额度、探测量、拦截错误、用户可见错误。
- 运营健康分：按请求成功率、单位账号统计源健康率、渠道健康率、今日成本快照覆盖度加权计算，用于综合页第一眼判断系统状态。
- 今日流水成本：按自然日统计平台收入、上游成本、预计利润、毛利率、快照覆盖度和估算状态。
- 用户：总数、启用、停用、管理员、当前时间范围活跃用户、总额度、已用额度、累计请求、Top 用户调用排行。
- 渠道：总数、启用、手动停用、自动停用、健康、降级、异常、已用额度。
- 渠道活动：当前时间范围内按请求量排序的 Top 渠道、额度和 Token。
- SFT：当前时间范围内的 20、50、51、52、29、59、2、21、5 计数和容错成功率。
- 模型活动：当前时间范围内按请求量排序的模型请求、额度和 Token。
- 趋势序列：按时间桶返回请求数、成功数、错误数、探测数、Token、额度、成功率，用于主题页的趋势条形图。
- 模型完整排行：按模型聚合请求数、成功数、错误数、直连成功 `2`、容错成功 `20`、容错失败 `50`、旧错误 `5`、流式数、Token、额度、平均耗时、成功率、流式占比。
- 用户消耗排行：按用户聚合请求、成功、错误、类型拆分、流式、Token、额度、平均耗时、成功率。
- 渠道调用排行：按渠道聚合请求、成功、错误、类型拆分、流式、Token、额度、平均耗时、成功率，并补齐渠道名称、启用状态、健康状态、优先级、权重、单位 ID、单位名称、单位类型、账号 ID 和账号显示名。
- 多维度钻取排行：支持 `dimension=group|token|endpoint|ip`，按分组、令牌、请求路径/IP 统计请求、成功/失败、类型拆分、流式、Token、额度、平均耗时、成功率。
- 模型覆盖与缺口矩阵：按模型汇总 `abilities` 覆盖分组、启用渠道、健康/降级/停用渠道，并叠加当前时间范围内该模型的真实请求、错误、Token、额度和平均耗时，用于发现“模型有请求但覆盖差”“模型只剩少量健康渠道”“模型没有启用渠道”等风险。
- 流水成本明细：支持 `dimension=unit|channel|model|user`，返回收入、上游成本、预计利润、毛利率、请求数、Token、额度、快照数量、成本口径和估算状态。
- 流水成本明细在 `dimension=channel` 时必须补齐渠道所属单位/账号、渠道状态、健康状态、优先级、权重；在 `dimension=unit` 时必须补齐单位类型和账号显示名。

接口实现要求：

1. 不能由前端拉取大量明细再聚合。
2. 聚合 SQL 必须兼容 SQLite、MySQL、PostgreSQL。
3. 独立统计块可以并发查询，避免远程数据库延迟累加。
4. 统计接口失败不能影响单位管理基础配置页面。
5. `hours` 参数必须有上限，当前最大 30 天，防止误扫过大日志范围。
6. 看板统计口径必须避免重复计算链路子日志：用户可见请求按 `2/20/50/5` 统计，费用和 Token 只按 `2/20` 成功消费统计，`21/51/52/29/59` 主要进入 SFT 和质量辅助指标。
7. 如果未配置 `LOG_SQL_DSN`，日志库复用业务库，统计 SQL 的数据库方言必须按业务库实际类型推断，不能继续使用 `common.LogSqlType` 的默认 SQLite 值。
8. 趋势分桶字段必须显式 cast 为整数，避免 MySQL 将桶表达式返回 decimal 字符串导致扫描失败。
9. 主题页完整排行必须由后端分页聚合返回，不能由前端拿 `overview` 的 Top 8 当完整排行。
10. 排行排序参数只能使用白名单：`requests`、`tokens`、`quota`、`errors`、`latency`、`success`，不能把前端参数直接拼进 SQL。
11. 今日流水成本中，平台收入按今日 `2/20` 成功消费日志的 `quota / QuotaPerUnit` 换算 USD。
12. 今日上游成本按单位账号监控快照表计算：同一统计源今日最新 `used_amount` 减今日第一条 `used_amount`，只统计 USD 快照且负差值归零。
13. 上游成本需要今日至少两次成功检测快照；不足两次时 `estimated=true`，页面显示估算和快照覆盖度。
14. 每次单位账号监控成功检测后必须写入 `unit_account_monitor_snapshots`，用于后续流水成本、余额趋势和可用天数估算。
15. 成本明细必须区分“精确成本”和“分摊成本”：
    - `dimension=unit`：上游成本按单位账号统计源今日首尾快照差值计算，属于精确快照口径；如果快照不足两次，则 `estimated=true`。
    - `dimension=channel`：渠道收入来自该渠道今日 `2/20` 成功消费日志；上游成本按同一单位账号下各渠道成功消费额度比例分摊，属于估算口径。
    - `dimension=model`：模型收入来自该模型今日 `2/20` 成功消费日志；上游成本按模型成功消费额度占全站成功消费额度比例分摊，属于估算口径。
    - `dimension=user`：用户收入来自该用户今日 `2/20` 成功消费日志；上游成本按用户成功消费额度占全站成功消费额度比例分摊，属于估算口径。
16. 成本接口必须通过 `cost_source` 明确口径：
    - `snapshot_delta`：单位账号快照差值。
    - `allocated_by_quota`：按成功消费额度分摊。
    - `missing_monitor`：渠道关联的单位账号没有统计源或缺少可用快照。
    - 前端必须同时展示 `estimated`、`baseline_ready`、`platform_status`、`snapshot_count`、`first_snapshot_time`、`last_snapshot_time`，不能只显示一个总金额。
17. 多维度钻取排行必须使用同一套用户可见日志口径：
    - 请求数按 `2/20/50/5` 统计。
    - Token 和额度只按 `2/20` 成功消费统计。
    - 错误数按 `50/5` 统计。
    - 分组字段使用日志库专用 `logGroupCol`，避免 MySQL/PostgreSQL/SQLite 对 `group` 保留字处理不一致。
18. 运营健康分权重：
    - 请求成功率：45%。
    - 单位账号统计源健康率：20%。
    - 渠道健康率：20%。
    - 今日成本快照覆盖率：15%。
    - 总分低于 60% 为 `critical`，低于 80% 为 `warning`，其余为 `healthy`。
19. 模型覆盖与缺口矩阵必须以后端结构化数据为准：
    - 覆盖来源是 `abilities` 表，不从日志排行反推。
    - 渠道状态来源是 `channels.status` 与 `channels.health_status`。
    - 请求、错误、Token、额度和平均耗时来源是当前时间范围内 `logs` 的用户可见口径。
    - 风险等级必须明确 `critical`、`warning`、`idle`、`ok`，页面不能只显示数字让管理员自行猜。

## 五个主题页口径

### 综合

目标是给管理员一个默认态势页，回答“当前系统整体是否正常、是否在消耗、是否有异常”。展示：

- 请求质量：请求数、成功率、平均耗时、流式占比。
- 运营健康分：请求成功率、统计源健康率、渠道健康率、成本覆盖率的加权总分。
- 消耗统计：Token、额度、探测、拦截、用户可见错误。
- 统计源与余额：单位账号监控数量、正常/异常/待检测、当前余额、已用。
- 用户、渠道、模型、SFT 摘要：Top 用户、Top 模型、Top 渠道、20/50/51/52/29/59。

### 模型&性能

目标是回答“哪些模型被调用、调用是否稳定、性能是否退化”。展示：

- 模型活动数量、Top 模型请求、Token、额度。
- 模型完整排行：请求、成功/失败、类型拆分、成功率、流式占比、平均耗时、Token、额度。
- 模型覆盖与缺口矩阵：模型覆盖分组、启用渠道、健康渠道、降级渠道、停用渠道、覆盖率、健康率、最近请求和风险原因。
- 请求趋势、成功趋势、错误趋势。
- 平均耗时、成功率、流式占比。
- 后续应扩展模型级 P50/P90/P95/P99、首字耗时、模型-渠道矩阵。

### 用户&消耗

目标是回答“谁在用、用掉多少、错误是否影响用户”。展示：

- 用户总数、启用/停用、管理员、当前范围活跃用户。
- Token 趋势、额度趋势、请求趋势。
- Top 用户请求、Token、额度排行。
- 用户完整排行：请求、成功/失败、类型拆分、成功率、流式占比、平均耗时、Token、额度。
- 用户可见错误和拦截错误。
- 多维度钻取表默认可切换到令牌排行，用于定位某个下游 token 的请求、费用、错误率和流式占比。
- 后续应扩展大户异常增长、用户失败率排行、令牌级模型拆分、令牌级每日趋势。

### 流水&成本

目标是回答“今天平台实际流水、上游成本和预计利润是多少”。展示：

- 今日流水成本：平台收入、上游成本、预计利润、毛利率、快照覆盖度。
- 平台收入：今日 `2/20` 成功消费日志的 `quota / QuotaPerUnit`。
- 上游成本：同一单位账号统计源今日最新快照 `used_amount` 减今日首个快照 `used_amount`。
- 利润口径：平台收入减上游成本；快照不足两次时显示为估算。
- 流水成本明细表：可切换 `单位/账号`、`渠道`、`模型`、`用户` 四个维度，并按收入、成本、利润、毛利率、请求、Token、额度排序。
- 流水成本明细表必须把“对象”和“归属”分开：对象是单位/渠道/模型/用户本身，归属是单位、账号、渠道优先级/权重等来源信息。
- 余额和已用必须作为独立列展示，方便判断上游成本差值是否和当前余额状态一致。
- 成本口径必须显示 `快照差值`、`额度分摊` 或 `缺少统计源`，并显示精确/估算状态。
- 趋势辅助：收入额度趋势、成功请求趋势、错误影响趋势。
- 同页展示单位账号统计源表，方便直接检查今日上游成本是否因为快照不足、统计源异常或余额单位非 USD 而处于估算状态。
- 后续应扩展小时级流水、单位成本排行、模型利润排行、用户利润贡献排行。

### 页面展示规则

1. 每个主题页顶部必须有标题、当前统计范围、生成时间、主题焦点，不允许只放一排含义不明的 Tab。
2. 核心 KPI 卡片必须显示数值、辅助口径、状态标签；有比例含义的指标使用进度条显示覆盖度或成功率。
3. `流水&成本` 独立成页，不混在综合页里。综合页只保留全局态势，成本页负责收入、成本、利润和快照覆盖。
4. 模型、用户、渠道主题页必须有后端分页排行表；Top 摘要只做预览，不能替代完整排行。
5. 单位账号统计源表可以出现在 `综合`、`流水&成本`、`单位&渠道` 三个主题页：
   - `综合`：用于全局追溯。
   - `流水&成本`：用于检查成本快照覆盖。
   - `单位&渠道`：用于检查上游账号健康与采集能力。
6. 页面文案必须解释数据口径，尤其是 `2/20/50/5` 用户可见日志、`21/51/52/29/59` 链路辅助日志、平台收入和上游成本的计算差异。

### 单位&渠道

目标是回答“上游单位/账号/渠道是否健康，余额和采集能力如何”。展示：

- 统计源总数、启用、正常、异常、待检测。
- 余额汇总、已用、Token/Model/Group 采集能力。
- 渠道启用/总数、健康/降级/异常、Top 渠道请求、Token、额度。
- 渠道完整排行：请求、成功/失败、类型拆分、成功率、流式占比、平均耗时、Token、额度，并显示渠道启用状态、健康状态、优先级、权重、所属单位、单位类型和所属账号。
- 渠道请求趋势、探测趋势、错误趋势。
- 多维度钻取表默认可切换到分组排行，也可切换端点/IP，用于定位某个分组、路径或来源 IP 的请求和错误。
- 顶部侧栏面板必须展示当前渠道排行前几名、统计源采集状态、渠道配置状态和当前钻取维度的前几名，避免只有表格没有运营摘要。
- 后续应扩展单位账号余额预计可用天数、渠道连续失败、分组内可用覆盖率。

## 数据一致性要求

1. 单位和账号是配置主体。
2. 单位账号统计源是监控主体。
3. 统计源可以从单位管理内创建，但不反向修改单位账号的基础配置。
4. 统计详情从实际检测结果生成，不能手工伪造余额、模型、令牌等监控值。
5. 老数据如果缺少单位或账号关系，页面需要正常显示 `-` 或统计源已有快照，不能白屏或报错。

## 渠道与单位账号联动

渠道创建和编辑页面需要把单位配置真正接入渠道配置闭环，而不是只做展示字段。

字段顺序固定为：

```text
所属单位 -> 所属账号 -> API 地址 -> 密钥 / 获取平台令牌 -> 类型 -> 名称 -> 其他渠道配置
```

交互规则：

1. 所属单位下拉只读取启用单位，显示单位名称、平台类型、挂载账号数和有效 API 地址。
2. 选择所属单位后：
   - 清空旧的所属账号。
   - 清空旧的平台令牌列表、已选平台令牌和令牌选项。
   - 读取该单位下的挂载账号。
   - 如果单位有有效 API 地址，自动回填渠道 `base_url`，但允许管理员继续手工编辑。
3. 选择所属账号后：
   - 再次按单位有效 API 地址回填 `base_url`。
   - 清空旧的平台令牌缓存，防止把上一个账号的令牌填进当前渠道。
4. 密钥字段旁的按钮命名为“获取平台令牌”。
   - 这里获取的是所属单位平台里的 API 令牌。
   - 不是 NACP 当前站点的用户令牌。
   - 不是单位账号自身的访问令牌。
5. 支持平台令牌管理的单位类型可以读取、选择、新建平台令牌，并把选中的令牌 key 填入渠道密钥。
6. 不支持平台令牌管理的平台必须返回明确错误，例如“当前单位类型 xxx 不支持平台令牌管理”，前端必须直接展示后端 message，不能只显示泛化 Axios 错误。
7. 后端保存渠道时必须校验：
   - `unit_id` 和 `unit_account_id` 不能为负数。
   - 如果传入 `unit_account_id`，必须确认账号存在。
   - 如果同时传入 `unit_id` 和 `unit_account_id`，必须确认账号属于该单位。
   - 如果只传入 `unit_account_id`，后端自动归一化 `unit_id = account.unit_id`。
   - 如果传入不存在的单位或账号，返回业务错误，不 panic。
8. 编辑渠道时如果密钥为空字符串，不能覆盖原有密钥；这是原有渠道编辑安全语义，单位联动不能破坏。

2026-05-18 本地闭环验证：

- `/api/unit` 和 `/api/unit/` 均需要兼容。
- `/api/unit/:id/accounts` 能返回挂载账号。
- `/api/unit/:id/accounts/:account_id/token_options` 能返回平台分组和模型。
- `/api/unit/:id/accounts/:account_id/tokens` 能返回平台令牌。
- 新建禁用测试渠道后，`channels.unit_id`、`channels.unit_account_id`、`base_url`、`models`、`group` 保存正确。
- 编辑该测试渠道后，单位和账号关联继续保持。
- 使用不存在的 `unit_account_id` 创建渠道时返回“所属账号不存在”。
- 请求体缺少 `channel` 时必须返回 `channel cannot be empty`，不能触发空指针 panic。

## 后续计划

1. 增加用户统计细分：大户用量、消费排行、异常用户、分组分布。
2. 增加模型统计细分：成功率、费用、首字延迟、流式占比、模型-渠道矩阵。
3. 增加渠道健康细分：错误率、延迟、连续失败次数、分组内可用渠道覆盖率。
4. 增加 SFT 细分：51/52 错误原因、链路路径、最终渠道分布、分组成功率。
5. 引入 Sub2API 风格的 `raw / preagg / auto` 查询模式，当前先用 raw，后续切小时级预聚合。
6. 引入 Metapi 风格的聚合水位和可重算投影器，避免日志量变大后每次扫 `logs`。
7. 不同平台类型的采集逻辑继续分目录维护，例如 `newapi`、`sub2api`、`oneapi` 等，不混写平台差异。
8. 单位平台适配层必须执行 `.ai/L3#Standards/standards/01.arch-03.unit-platform-adapter.md`：平台 adapter 独立，协议级工具复用。已经明确有独立名称和接口特征的平台，例如 `rixapi`、`shellapi`、`veloera`、`onehub`、`donehub`、`anyrouter`、`cliproxyapi`、`geminicli`、`antigravity`、`codex`，都必须独立目录和 adapter，不能因为当前兼容 NewAPI/OneAPI/OpenAI-compatible 就糅合进母类型。
9. 单位平台令牌能力也必须走 adapter：`FetchTokens`、`FetchTokenOptions`、`CreateToken` 不能继续在 `service/unit_tokens.go` 用平台类型 switch 硬编码。`service/unit_tokens.go` 只保留业务入口和 `model.Unit` / `model.UnitAccount` 到 `unitplatform.Credentials` 的转换。
10. Sub2API 不能复用 NewAPI/OneAPI 面板接口。Sub2API 适配器必须走 `/api/v1/auth/me`、`/api/v1/keys`、`/api/v1/api-keys`、`/api/v1/groups`、`/v1/models` 等独立接口；余额按 `/api/v1/auth/me` 暴露的 USD balance 展示，令牌创建使用 Sub2API 自己的 `POST /api/v1/keys` / `POST /api/v1/api-keys`。
11. API Key 直连平台只支持模型读取和连通性校验，不支持平台令牌管理和账号余额快照。当前包括 `openai`、`claude`、`gemini`、`cliproxyapi`。这些平台的 `FetchTokenOptions` 可以返回模型列表，但 `FetchTokens` 和 `CreateToken` 必须明确返回“不支持平台令牌管理”。
12. OAuth/CLI 直连平台也必须独立适配，不允许糅合进 OpenAI-compatible 或 Gemini 母类型：
    - `codex`：使用 `GET <base>/models?client_version=1.0.0`，带 `Authorization: Bearer <access_token>`、`Originator: codex_cli_rs`，如配置账号 ID 则带 `Chatgpt-Account-Id`。只读取模型，不支持余额快照、平台令牌读取和创建。
    - `geminicli`：使用 Google Service Usage 接口校验项目是否启用 `cloudaicompanion.googleapis.com`，校验通过后返回 Gemini CLI 当前可用静态模型清单。账号 ID 字段在该类型中表示 Google Cloud 项目 ID。
    - `antigravity`：使用 `POST /v1internal:fetchAvailableModels` 读取模型，优先使用单位 API 地址，并 fallback 到 `cloudcode-pa.googleapis.com`、`daily-cloudcode-pa.googleapis.com`、`daily-cloudcode-pa.sandbox.googleapis.com`。账号 ID 可作为 project 字段。
13. 截至 2026-05-18，已拆分并接入的单位平台类型包括：`newapi`、`rixapi`、`shellapi`、`oneapi`、`oneapifork`、`veloera`、`onehub`、`donehub`、`anyrouter`、`sub2api`、`openai`、`claude`、`gemini`、`geminicli`、`antigravity`、`cliproxyapi`、`codex`。如果后续从参考项目或实际运营中发现新的独立平台名称或接口特征，继续新增独立 adapter。
