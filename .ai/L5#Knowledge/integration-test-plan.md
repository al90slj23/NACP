## 测试场景矩阵

### 一、基础重试功能

| # | 场景 | 渠道配置 | 预期结果 | 验证点 |
|---|------|---------|---------|--------|
| 1.1 | 所有渠道正常 | 只保留 C/D/E/F | 客户端成功，走最高优先级 | 无重试发生 |
| 1.2 | 最高优先级全坏 | A+B(坏) + C/D(好) | 客户端成功，切到 P50 | 日志显示重试路径 |
| 1.3 | 所有渠道全坏 | 只有 A+B | 客户端收到错误 | 错误正确返回 |
| 1.4 | 同优先级多个坏渠道 | A+B(坏,P100) + C(好,P50) | 尝试 A→B→C 成功 | 同优先级排除生效 |
| 1.5 | 同渠道重试成功 | 用限流渠道（偶发 429） | 第 2-3 次重试成功 | 同渠道重试计数 |

### 二、健康状态机

| # | 场景 | 操作 | 预期结果 | 验证点 |
|---|------|------|---------|--------|
| 2.1 | 401 不改健康状态 | 发请求到假 key 渠道 | health_status 保持 healthy | DB 查询确认 |
| 2.2 | 401+禁用关键词→Degraded | 模拟"quota exceeded"错误 | 立即变 Degraded | DB 查询确认 |
| 2.3 | 502/503→Probing→Degraded | 连续 2 次 5xx 错误 | Probing→Degraded | 状态转换日志 |
| 2.4 | Degraded 渠道被跳过 | 手动设 health_status=degraded | 请求不发到该渠道 | use_channel 不含该 ID |
| 2.5 | 恢复流程 | Degraded→周期探测成功 3 次→Recovering | 状态正确转换 | DB 查询 |
| 2.6 | Recovering 期间出错→回退 | Recovering 状态下发失败请求 | 回到 Degraded | 状态转换日志 |
| 2.7 | 手动禁用免疫 | ManuallyDisabled 渠道 | 不做任何自动转换 | 状态不变 |

### 三、渠道亲和性 + 健康感知

| # | 场景 | 配置 | 预期结果 | 验证点 |
|---|------|------|---------|--------|
| 3.1 | 亲和渠道健康→正常亲和 | 亲和绑定到 C | 请求发到 C | channel_id=C |
| 3.2 | 亲和渠道 Degraded→跳过 | 亲和绑定到 C，C 设为 Degraded | 请求发到其他渠道 | channel_id≠C |
| 3.3 | 跳过后 SwitchOnSuccess | 亲和 C(Degraded)→选到 D→成功 | 亲和缓存更新到 D | 下次请求到 D |
| 3.4 | SkipRetry=true + Healthy 失败 | 亲和渠道偶发失败 | 报错给客户端（不切换） | 错误返回 |
| 3.5 | SkipRetry=true + Degraded | 亲和渠道已 Degraded | 跳过亲和，正常选择 | 不报错 |

### 四、并行预热

| # | 场景 | 配置 | 预期结果 | 验证点 |
|---|------|------|---------|--------|
| 4.1 | 预热成功→快速切换 | A(坏)+C/D(好) | 切到 C/D 时无额外延迟 | 响应时间对比 |
| 4.2 | 预热全失败 | A(坏)+B(坏,P50) | 最终报错 | 错误返回 |

### 五、错误分类

| # | 场景 | 错误码 | 预期行为 |
|---|------|--------|---------|
| 5.1 | 400 Bad Request | 400 | 直接切渠道，不改健康状态，不同渠道重试 |
| 5.2 | 401 Unauthorized | 401 | 切渠道，不改健康状态（无禁用关键词时） |
| 5.3 | 401 + "quota exceeded" | 401 | 立即 Degraded |
| 5.4 | 429 Rate Limited | 429 | 同渠道重试 + 切渠道 |
| 5.5 | 500 Internal Error | 500 | 同渠道重试 + 切渠道 + 健康状态变更 |
| 5.6 | 502 Bad Gateway | 502 | 同渠道重试 + 切渠道 + 健康状态变更 |
| 5.7 | 503 Service Unavailable | 503 | 同渠道重试 + 切渠道 + 健康状态变更 |
| 5.8 | 504 Gateway Timeout | 504 | 不重试，不改健康状态 |

### 六、边界情况

| # | 场景 | 预期结果 |
|---|------|---------|
| 6.1 | 分组内只有 1 个渠道且坏 | 报错给客户端 |
| 6.2 | 分组内无渠道 | 报"无可用渠道"错误 |
| 6.3 | 所有渠道 Degraded + fallback | fallback 生效，用 Degraded 渠道 |
| 6.4 | 并发请求同时触发状态转换 | 无 race condition |
| 6.5 | 流式请求重试 | 窗口 1 错误正常重试 |
| 6.6 | 大请求体重试 | body 正确重放 |
| 6.7 | RetryTimes=0 时的行为 | 同渠道重试仍生效，跨优先级不生效 |

### 七、日志与可观测性

| # | 验证点 |
|---|--------|
| 7.1 | 每次渠道错误都记录到 error log |
| 7.2 | 成功请求记录 use_channel 路径 |
| 7.3 | 健康状态转换有 SysLog |

## 需要的资源

### 已有
- 4 个真实 API key（CCM, AWS, Kiro, Claude）
- 假 key（自己造）
- 测试服务器 + NACP 部署

### 还需要
1. **一个能返回 429/500/502/503 的渠道** — 用于测试非 401 错误的重试行为。可以通过以下方式模拟：
   - 配置一个 base_url 指向不存在的服务器（会返回 502/503）
   - 或者提供一个已知会限流的 key（触发 429）
   - 或者我在测试服务器上搭一个 mock server 返回指定错误码

2. **确认是否需要测试亲和性场景（3.1-3.5）** — 这需要用 Claude CLI 或 Codex 发请求来触发亲和规则（因为规则匹配 model_regex 和 path_regex）

## 测试执行方式

1. 通过 API 动态调整渠道配置（增删改渠道、修改优先级/权重）
2. 通过 API 发送测试请求
3. 通过 API + DB 查询验证结果
4. 全部自动化脚本执行
