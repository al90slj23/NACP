# NACP统计主线阶段总结与续接计划

> 最后更新：2026-05-18

## 当前主线位置

NACP统计主线已经完成从“单位监控列表”到“管理员运营统计看板”的第一阶段改造。当前页面固定为五个主题页：

- 综合
- 模型&性能
- 用户&消耗
- 流水&成本
- 单位&渠道

最近一轮重点完成了 `单位&渠道` 与 `流水&成本` 的数据深化。当前主线可以暂时挂起，先处理两个较小支线任务；支线完成后，应从本文的“回到主线后的执行顺序”继续。

## 最近完成的工作

### 1. 单位管理与单位账号统计

- 新增“单位”概念，用于承载上游资源平台。
- 单位下可挂载账号，账号字段包括账号、密码、账户访问令牌、账户 ID。
- 单位管理列表新增详情入口。
- 单位详情会读取单位基础信息、挂载账号、账号统计源和检测结果。
- 单位账号统计源支持读取余额、已用、模型、令牌、分组、上游用户、平台状态等信息。
- NewAPI 兼容采集器会读取 `/api/status`，并提取系统名称、版本、额度单位、`quota_per_unit`、`price`、`stripe_unit_price`、`usd_exchange_rate`、`custom_currency_symbol`、`custom_currency_exchange_rate`、`top_up_link` 等字段。
- `price` 当前作为充值倍率展示；没有上游字段时显示 `-`，不能推断。

### 2. 平台类型适配器

已建立 `service/unitplatform/` 适配器体系。原则是：

- 已知独立平台类型单独建目录，不和母类型糅合。
- 可复用底层 helper，但外部 adapter 入口必须独立。
- 平台识别、凭据要求、监控、令牌、模型、分组、计价信息都按平台类型隔离。

当前已拆分的平台类型包括：

- `newapi`
- `rixapi`
- `shellapi`
- `oneapi`
- `oneapifork`
- `veloera`
- `onehub`
- `donehub`
- `anyrouter`
- `sub2api`
- `openai`
- `claude`
- `gemini`
- `geminicli`
- `antigravity`
- `cliproxyapi`
- `codex`

### 3. 渠道与单位账号联动

渠道新建/编辑流程已经接入单位配置闭环：

```text
所属单位 -> 所属账号 -> API 地址 -> 密钥 / 获取平台令牌 -> 类型 -> 名称 -> 其他渠道配置
```

当前规则：

- 选择所属单位后读取该单位下账号。
- 选择所属单位后默认回填单位有效 API 地址。
- 选择所属账号后可以读取该平台账号下的平台令牌。
- “获取平台令牌”指的是上游平台内的 API 令牌，不是 NACP 本站用户令牌。
- 后端保存渠道时校验 `unit_id`、`unit_account_id` 是否存在且匹配。
- 如果只传 `unit_account_id`，后端会归一化 `unit_id = account.unit_id`。
- 编辑渠道时空密钥不能覆盖原密钥。

### 4. NACP统计看板

已新增专用聚合接口：

- `/api/nacp_stats/overview`
- `/api/nacp_stats/models`
- `/api/nacp_stats/users`
- `/api/nacp_stats/channels`
- `/api/nacp_stats/costs`
- `/api/nacp_stats/dimensions`
- `/api/nacp_stats/model_coverage`

当前统计口径：

- 用户可见请求按 `2/20/50/5` 统计。
- 成功消费、Token、额度只按 `2/20` 统计。
- `21/51/52/29/59` 作为链路辅助日志，进入 SFT 和质量辅助指标，不进入用户收入。
- 今日平台收入 = 今日 `2/20` 成功消费日志 `quota / QuotaPerUnit`。
- 今日上游成本 = 同一单位账号统计源今日最新快照 `used_amount` - 今日首个快照 `used_amount`。
- 快照不足两次时，成本标记为估算。

### 5. 单位&渠道深化

最近补齐：

- 渠道排行返回单位名称、单位类型、账号显示名。
- 渠道排行显示渠道启用状态、健康状态、优先级、权重。
- 页面新增 `所属单位/账号` 列。
- 顶部摘要面板展示渠道排行预览、统计源采集状态、渠道配置状态、当前钻取维度前几名。
- 多维度钻取支持分组、令牌、端点、IP。

### 6. 流水&成本深化

最近补齐：

- 成本明细支持 `dimension=unit|channel|model|user`。
- `dimension=unit` 使用单位账号快照差值，属于精确口径。
- `dimension=channel` 按同一单位账号下各渠道成功消费额度比例分摊上游成本，属于估算口径。
- `dimension=model` 按模型成功消费额度占全站成功消费额度比例分摊上游成本。
- `dimension=user` 按用户成功消费额度占全站成功消费额度比例分摊上游成本。
- 明细表新增“对象”和“归属”拆分。
- 明细表新增余额/已用列。
- 成本口径显示 `快照差值`、`额度分摊`、`缺少统计源`，并显示精确/估算、平台状态。

## 回到主线后的执行顺序

### P0：真实消费日志闭环验证

目的：验证 `单位&渠道` 和 `流水&成本` 在真实 `2/20/50` 日志存在时是否按预期变化。

执行内容：

1. 使用本地开发环境 `http://localhost:23901/` 和本地后端 `http://localhost:23900/`。
2. 通过 API 创建普通用户。
3. 普通用户创建 token。
4. 配置可控渠道和真实渠道，确保渠道绑定单位与单位账号。
5. 发起真实模型请求，生成至少以下日志：
   - `2`：直连成功消费。
   - `20`：容错后成功。
   - `50`：容错后失败。
   - 必要时生成 `51/52/21/29/59` 作为链路辅助日志。
6. 刷新 `/console/nacp_stats`。
7. 检查：
   - 渠道排行是否出现真实渠道。
   - 渠道归属单位/账号是否正确。
   - 成功/失败、类型拆分、Token、额度是否正确。
   - `流水&成本` 的收入是否按 `2/20` 增加。
   - `dimension=channel` 是否按渠道额度分摊上游成本。
   - `dimension=model` 和 `dimension=user` 是否按额度比例分摊。
   - `dimensions` 的分组/令牌/端点/IP 是否出现真实排行。

放行标准：

- 所有接口返回成功。
- 页面不白屏、不报业务 500。
- 空数据和真实数据都显示合理。
- 成本口径能明确区分精确和估算。
- `21/51/52/29/59` 不污染用户收入、Token、额度统计。

### P1：余额与成本趋势

目的：把单位账号快照从“今日首尾差值”升级为可观察趋势。

计划内容：

- 新增或扩展接口，按小时/天返回单位账号快照趋势。
- 显示余额变化、已用变化、上游成本变化。
- 支持单位、账号、平台类型维度筛选。
- 计算预计可用天数。
- 标记快照缺失、余额单位非 USD、使用量回退等异常。

数据口径：

- 原始数据来源：`unit_account_monitor_snapshots`。
- 成本趋势：相邻快照 `used_amount` 差值，负值归零并标记异常。
- 余额趋势：直接使用 `current_balance`。
- 可用天数：最近窗口平均日消耗为分母，余额为分子；消耗为 0 时显示 `-`。

### P1：单位&渠道风险视图

目的：让管理员能直接看到“哪些渠道/单位配置存在风险”，而不是自己从表里找。

计划内容：

- 分组内可用渠道覆盖率。
- 有流量但未绑定单位账号的渠道。
- 有单位账号但没有统计源的渠道。
- 连续失败渠道排行。
- 健康但无流量渠道。
- 单位账号异常影响了哪些渠道。
- 渠道优先级/权重和实际流量是否匹配。

### P1：流水&成本利润视图

目的：让成本页从“收支流水”继续升级为“利润分析”。

计划内容：

- 模型利润排行。
- 用户利润排行。
- 渠道毛利率排行。
- 单位账号成本排行。
- 收入有记录但无法归属成本的异常列表。
- 成本有增长但没有对应收入的异常列表。

### P2：前端噪音警告清理

当前仍存在一些非本轮新增导致的开发期噪音：

- Semi UI / React StrictMode 的 `findDOMNode` 警告。
- `ReactDOM.render` 兼容警告。
- 个别组件 prop 类型警告。

这些警告目前不阻塞 NACP统计主线，但会干扰后续判断真正错误。建议单独开一轮清理，不和统计口径改造混在一起。
